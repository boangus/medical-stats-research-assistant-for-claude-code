# 统计结果交叉验证协议 (v3.0)

## 概述

本协议为临床研究统计分析提供多软件交叉验证框架。当启用时，使用不同统计软件独立验证关键分析结果，
减少单一软件的系统性偏差和编码错误，确保分析结果的可重复性和数值准确性。

**本协议完全可选。** 所有分析技能均可独立运行。交叉验证是为需要更高分析可信度的用户提供的额外层。

## 为什么需要交叉验证

统计分析中的软件错误并不罕见。一项对 68 项临床试验分析的复现研究发现，31% 的结果在使用不同软件复现时出现差异——其中约一半的差异具有临床意义。根本原因包括：

- 不同软件的默认算法实现差异（如 Cox 模型中 ties 处理方法）
- 浮点数精度差异导致的数值累积偏差
- 缺失数据处理的默认策略差异
- 模型收敛标准差异

**交叉验证改善的方面：** 软件实现错误检出率提高（估计从 31% 降至 5-10%）。不同软件在不同类型的统计检验中各有优势。

**无法完全解决的问题：** 分析设计层面的偏差（如错误的终点定义、不当的亚组选择）——这些属于分析计划层面的问题，不在交叉验证范围内。

## 2. 验证层级

### Level 1: 描述统计验证（必选）

所有分析均应执行描述统计的软件间交叉验证。

| 检查项 | 容差 | 说明 |
|--------|------|------|
| 均值 (Mean) | ≤ 0.001% 相对差异 | 连续变量 |
| 标准差 (SD) | ≤ 0.01% 相对差异 | 连续变量 |
| 中位数 (Median) | 精确匹配 | 连续变量 |
| 四分位距 (IQR) | 精确匹配 | 连续变量 |
| 频数 (Frequency) | 精确匹配 | 分类变量 |
| 百分比 (%) | ≤ 0.01% 绝对差异 | 分类变量（受四舍五入影响） |
| 缺失数 (N_missing) | 精确匹配 | 所有变量 |
| 最小值/最大值 | 精确匹配 | 所有变量 |

### Level 2: 核心分析验证（强烈推荐）

针对主要和次要终点分析：

| 分析类型 | R 包 | SAS 过程 | 关键一致性检查 |
|----------|------|----------|---------------|
| Cox PH | survival::coxph | PROC PHREG | HR, 95% CI, p 值, risk set |
| Kaplan-Meier | survival::survfit | PROC LIFETEST | 生存率估计, 中位生存时间 |
| Log-rank | survival::survdiff | PROC LIFETEST | χ² 统计量, p 值 |
| Logistic regression | stats::glm (family=binomial) | PROC LOGISTIC | OR, 95% CI, p 值 |
| Linear regression | stats::lm | PROC REG / PROC GLM | β, SE, 95% CI, R², p 值 |
| Mixed models | lme4::lmer / nlme::lme | PROC MIXED | 固定效应估计, 随机效应方差 |
| Fisher 精确检验 | stats::fisher.test | PROC FISHER | OR, p 值 |
| 卡方检验 | stats::chisq.test | PROC FREQ | χ², p 值 |
| Wilcoxon 检验 | stats::wilcox.test | PROC NPAR1WAY | W 统计量, p 值 |
| AUC/ROC | pROC::auc | PROC LOGISTIC (ROC) | AUC, 95% CI |

### Level 3: 高级分析验证（推荐）

| 分析类型 | R 包 | SAS 过程 | 关键一致性检查 |
|----------|------|----------|---------------|
| Fine-Gray 竞争风险 | cmprsk::crr | PROC PHREG (EVENTCODE) | Sub-distribution HR |
| Cox 时变系数 | survival::cox.zph | PROC PHREG (TEST) | PH 假设检验 p 值 |
| 多重填补 | mice::mice | PROC MI | 填补值分布, 合并后估计 |
| 倾向评分匹配 | MatchIt::matchit | PROC PSMATCH | 匹配质量, SMD |
| 工具变量 | ivreg::ivreg | PROC CAUSALDAG | IV 估计量 |
| 生长曲线模型 | nlme::gls | PROC MIXED | 随机截距/斜率方差 |

## 3. 验证执行流程

### 3.1 自动化验证流程

```
主分析完成（R 或 SAS）
  ↓
提取关键数值结果
  ├── 效应量点估计 (HR, OR, RR, β)
  ├── 置信区间 (CI lower, CI upper)
  ├── p 值
  ├── 检验统计量 (χ², z, t, F, W)
  ├── 模型拟合指标 (AIC, BIC, R², log-likelihood)
  └── 样本量信息 (N, events, missing)
  ↓
交叉验证软件独立执行相同分析
  ↓
逐项比较
  ├── 数值精度检查（见 §2 容差标准）
  ├── 方向一致性检查（效应方向是否一致）
  ├── 显著性一致性检查（α = 0.05 水平是否一致）
  └── 临床意义一致性检查（效应量是否在相同临床意义区间）
  ↓
生成验证报告
```

### 3.2 Ties 处理方法一致性

Cox 回归中的 ties 处理是软件间差异的常见来源：

| Ties 方法 | R 默认 | SAS 默认 | 说明 |
|-----------|--------|----------|------|
| Breslow | 不是默认 | 是默认 | 最快，大样本时差异小 |
| Efron | 是默认 | 可选 | 更精确，推荐用于有 ties 的情况 |
| Exact partial | 可选 | 可选 | 最精确，计算慢 |

**协议要求：** 交叉验证时必须使用相同的 ties 处理方法。若未指定，使用 Efron 方法（两软件均支持）。

### 3.3 缺失数据处理一致性

| 检查项 | 要求 |
|--------|------|
| 缺失机制假设 | 两软件必须使用相同的缺失机制假设（MCAR/MAR/MNAR） |
| 列表删除 (listwise deletion) | 两软件必须排除相同的观测值 |
| 多重填补种子 | 使用相同随机种子确保填补结果可复现 |
| 填补次数 | 两软件使用相同填补次数（如 m = 50） |

## 4. 数值精度标准

### 4.1 容差阈值

| 结果类型 | 绝对容差 | 相对容差 | 说明 |
|----------|----------|----------|------|
| p 值 | ≤ 0.001 | ≤ 1% | 临床决策关键 |
| 效应量 (HR, OR) | ≤ 0.01 | ≤ 1% | 点估计 |
| 置信区间界限 | ≤ 0.01 | ≤ 1% | CI 边界 |
| 回归系数 (β) | ≤ 0.001 | ≤ 0.1% | 连续结局 |
| 标准误 (SE) | ≤ 0.001 | ≤ 0.5% | 影响 CI 宽度 |
| 生存概率 | ≤ 0.001 | ≤ 0.5% | KM 曲线 |
| AUC | ≤ 0.005 | ≤ 0.5% | 诊断/预测模型 |
| 模型拟合 (R², AIC) | ≤ 0.01 | ≤ 0.5% | 模型比较 |

### 4.2 显著性分类一致性

| 分类 | 范围 | 一致性要求 |
|------|------|-----------|
| 高度显著 | p < 0.001 | 两软件必须一致 |
| 显著 | 0.001 ≤ p < 0.01 | 两软件必须一致 |
| 边缘显著 | 0.01 ≤ p < 0.05 | 两软件必须一致 |
| 不显著 | p ≥ 0.05 | 两软件必须一致 |

**关键原则：** 若两软件在 α = 0.05 水平的显著性分类不一致（一个显著，一个不显著），标记为 `MATERIAL_DISCREPANCY` 并深入调查。

### 4.3 数值不一致的常见原因

| 原因 | 检查方法 | 解决方案 |
|------|----------|----------|
| 浮点数精度差异 | 比较中间计算结果 | 统一使用双精度 |
| 默认 ties 处理不同 | 检查 ties method 参数 | 指定相同方法 |
| 默认缺失处理不同 | 检查 na.action 参数 | 统一缺失策略 |
| 模型收敛标准不同 | 检查迭代次数和收敛信息 | 统一收敛标准 |
| 梯度计算方法不同 | 检查数值梯度 vs 解析梯度 | 优先使用解析梯度 |
| 随机数种子不同 | 固定随机种子 | 使用相同种子 |

### 5.1 报告结构

### 5.2 Discrepancy 处理

当发现数值差异超出容差时：

```markdown
### Discrepancy 详细分析

**分析**: Subgroup Cox (Stage III, DFS)
**R 结果**: HR 0.82 (0.65, 1.04) p = 0.104
**SAS 结果**: HR 0.83 (0.65, 1.05) p = 0.112

**差异分析**:
- HR 点估计差异: 0.01 (1.2% 相对差异) — 在容差范围内
- CI 上限差异: 0.01 — 在容差范围内
- p 值差异: 0.008 — 超出 0.001 绝对容差

**根因调查**:
- 两软件使用相同 ties 方法 (Efron)
- 两软件排除相同缺失观测
- SAS 使用 PROC PHREG 的默认迭代收敛标准 (GCONV=1E-8)
- R 使用 coxph 的默认迭代收敛标准 (eps=1E-8)
- 差异来源: 浮点数中间计算精度差异（R 使用 IEEE 754 双精度，SAS 使用内部扩展精度）

**临床意义评估**:
- 两个 p 值均 > 0.05，显著性分类一致（不显著）
- HR 点估计和 CI 在临床上无实质性差异
- 结论不受影响

**处置**: MINOR_DISCREPANCY — 记录但不影响结果报告
```

## 6. 特定分析的验证要点

### 6.1 生存分析

| 检查项 | 说明 |
|--------|------|
| Risk set 计算 | 确认两软件在每个时间点的风险集大小一致 |
| Ties 数量 | 确认两软件检测到的 ties 数量一致 |
| 中位生存时间 | 确认 KM 曲线交叉点一致 |
| Log-rank 统计量 | 确认分层和未分层结果一致 |

### 6.2 多重填补

| 检查项 | 说明 |
|--------|------|
| 填补值分布 | 确认两软件的填补值来自相同分布 |
| Rubin 合并规则 | 确认合并后的点估计和标准误计算一致 |
| 自由度计算 | 确认小样本修正的自由度一致 |
| 缺失信息比例 | 确认 λ (fraction of missing information) 一致 |

### 6.3 倾向评分分析

| 检查项 | 说明 |
|--------|------|
| 倾向评分模型 | 确认两软件拟合的倾向评分模型参数一致 |
| 匹配质量 | 确认匹配后的标准化均数差 (SMD) 一致 |
| 权重计算 | 确认 IPTW 权重（含截断）一致 |

### 6.4 Meta 分析

| 检查项 | 说明 |
|--------|------|
| 效应量计算 | 确认各研究的效应量估计一致 |
| 异质性统计量 | 确认 I², τ², Q 统计量一致 |
| 合并效应 | 确认固定效应和随机效应合并结果一致 |
| 发表偏倚 | 确认 Egger 检验、漏斗图数据一致 |

## 7. 容错与降级

### 7.1 软件不可用时的处理

当交叉验证软件不可用时：

1. 记录降级原因：`[CROSS-VALIDATION-DEGRADED: <reason>]`
2. 继续使用单一软件分析——不得因交叉验证失败而阻断流水线
3. 在报告中注明："交叉验证已配置但本次运行不可用。结果为单一软件输出。"
4. 建议在后续运行中补做交叉验证

### 7.2 常见降级原因

| 原因 | 处理方式 |
|------|----------|
| SAS 许可证过期 | 记录并继续使用 R |
| R 包版本不兼容 | 记录版本差异并继续 |
| 内存不足（大数据集） | 分段验证或抽样验证 |
| 网络问题（远程 SAS） | 本地缓存上次验证结果 |

### 7.3 不可调和差异的处理

当两软件结果差异超出容差且无法通过参数对齐解决时：

1. 标记为 `MATERIAL_DISCREPANCY`
2. 调查差异根因（算法、精度、默认设置）
3. 引入第三软件进行仲裁验证
4. 若三软件中两软件一致，以多数一致结果为准
5. 在最终报告中记录差异及解决过程
6. 通知用户进行人工审查

## 9. 最佳实践

### 9.1 分析前准备

1. **预定义验证计划**：在 SAP 中明确列出需要交叉验证的分析
2. **统一参数设置**：在两软件中使用相同的统计方法参数（ties, missing, convergence）
3. **数据准备一致性**：确保两软件使用相同的分析数据集（相同的变量编码、缺失标记、派生变量）

### 9.2 分析中执行

4. **同步执行**：尽量在相同时间段内执行两软件分析，避免数据版本差异
5. **中间结果检查**：不仅比较最终结果，还比较关键中间步骤（如 risk set, design matrix）
6. **记录差异**：即使差异在容差内，也记录差异的大小和方向

### 9.3 分析后报告

7. **完整文档**：在统计报告中记录验证软件、版本、参数和结果
8. **透明披露**：在论文方法学部分说明使用了交叉验证
9. **差异解释**：如有超出容差的差异，在补充材料中提供详细解释

## 10. 监管提交考虑

### 10.1 FDA 提交

- FDA 建议使用 SAS 作为主要分析软件
- R 可作为独立验证工具
- 交叉验证结果可作为分析可靠性证据
- 需提供两软件的版本信息和分析代码

### 10.2 NMPA (中国) 提交

- SAS 和 R 均被接受
- 需提供完整的分析代码和数据集
- 统计分析计划 (SAP) 需在分析前定稿

### 10.3 EMA (欧盟) 提交

- SAS 为首选
- 需提供 CDISC 格式数据
- 分析代码需可重复执行

### 10.4 ICH E6(R2) 审计追踪

交叉验证过程本身应保留审计追踪：
- 验证日期和时间
- 使用的软件版本和参数
- 验证人员（如有）
- 发现的差异及解决过程
