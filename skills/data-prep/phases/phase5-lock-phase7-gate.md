### Phase 5: 数据库锁定

> 依据：《药物临床试验数据管理与统计分析计划指导原则》数据库锁定章节

数据审核是数据库锁定的前置条件。

**锁定流程**：
1. 确认所有数据质疑已解决
2. 确认所有方案偏离已记录
3. 确认盲态审核已完成（如适用）
4. 生成数据库锁定清单
5. 用户确认锁定
6. 记录锁定时间和版本

**注意事项**：
- 数据库锁定后禁止解锁和再锁定（仅限严重数据错误时启动正式解锁流程）
- 如需解锁，必须事先规定条件和流程，并记录解锁原因
- 记录锁定版本号和日期

> **🔴 [MANDATORY]** → 检查点表 #7（MANDATORY-M1）。数据库锁定前必须等待用户最终确认。

**数据库锁定异常处理**：

| 触发条件 | 一线处理 | 仍失败兜底 |
|---------|---------|-----------|
| 锁定后发现数据错误 | 评估错误对分析的影响程度 | 轻微错误记录偏差不解锁；严重错误走正式解锁流程并记录原因 |
| 用户要求解锁并再锁定 | 确认解锁理由是否充分（严重数据错误） | 解锁后修改 → 重新盲态审核 → 重新锁定，记录版本号 |

### Phase 7: 数据质量门闸（阻断式检查）

> 由 Pipeline Orchestrator 在 Stage 1.5 触发。此阶段仅执行 9 项阻断检查，**不修改任何数据或清洗脚本**。
> 完整检查清单见 Pipeline §3 Stage 1.5；此处定义本 Skill 的前置产物、输入映射和输出格式。
> 参考：shared/reproducibility/pipeline_auditor.py — 数据清洗审计追踪（优先使用）；shared/reporting-guidelines/quality_checklist.md — 清洗完整性检查

**前置产物**（Phases 1-5 输出，门闸检查消费）：

| # | 产物 | 来源 Phase | 门闸检查项 |
|---|------|-----------|-----------|
| 1 | 验证报告 + 清洗日志 | Phase 1 + 3 | □1 清洗日志完整 |
| 2 | 变量清单（原始+标识） | Phase 1 | □2 变量定义明确 |
| 3 | EDA 数据质量报告 | Phase 4 | □3 缺失机制评估 |
| 4 | 盲态审核记录 | Phase 4 | □4 盲态审核完成 |
| 5 | 数据库锁定记录 | Phase 5 | □5 数据库锁定确认 |
| 6 | 清洗后数据 | Phase 3 | □6 逻辑一致性验证 |
| 7 | 规范化日志 | Phase 2 Step 2.5 | □7 值规范化完成 |
| 8 | 清洗脚本 | Phase 3 | □8 可重复性 |
| 9 | 数据隐私合规报告 | Phase 1 (PHI检测) | □9 隐私合规完成 |

> **注意**：分析所需的衍生变量（如 BMI、年龄分组、Charlson 指数）不在 Stage 1 构造。构造逻辑在 SAP（Stage 2）中预定义，实际构造在 Stage 3（分析执行）按 SAP 执行。Stage 1 仅输出原始变量和清洗修正后的变量。

**quality-gate 模式执行逻辑**：

```
逐项检查 □1-□9（检查标准见 Pipeline §3）
  → 每项标记: ✅通过 / ❌未通过 / ⬜不适用
  → 统计通过数 → 按 Pipeline 判定规则输出结论
```

**判定规则**（与 Pipeline §3 Stage 1.5 一致）：

| 通过数 | 结果 | 后续操作 |
|--------|------|---------|
| 全部通过 (9/9) | ✅ 通过 | 进入 Stage 2 |
| 7-8 项通过 | ⚠️ 条件通过 | 提示用户，记录风险后进入 Stage 2 |
| ≤6 项通过 或 项目 5/6/7/9 未通过 | ❌ **阻断** | **强制退回 Stage 1 修订** |

**输出**: 数据质量门闸报告（`gate_report_stage_1_5.md`）

```markdown
# 数据质量门闸报告 — {日期}
## 检查结果
| # | 检查项 | 结果 | 问题描述 |
|---|--------|------|---------|
| 1 | 清洗日志完整 | ✅/❌ | — |
| ... | ... | ... | ... |
## 判定结论: ✅通过 / ⚠️条件通过 / ❌阻断
## 回退方案（如阻断）
```

## Mode

### guided（默认）
完整流程：验证 → 逐项讨论 → 值规范化(检测到变体时) → 执行 → EDA 质量检查 → 盲态审核 → 锁定 → 门闸

### quick
仅验证Critical问题，自动修复明显格式问题（含默认值规范化策略），跳过 EDA 质量检查，其余标记待手动处理

### quality-gate
仅执行数据质量门闸检查，不修改数据或脚本。由 Pipeline Orchestrator 在 Stage 1.5 触发。

## 反例与黑名单

> **以下行为必须避免**。违反任何一条将导致数据质量不可靠或分析结果有偏。
> 完整医学统计反模式目录参见：shared/anti-patterns/medical_stats_anti_patterns.md（C1）

### 🚫 验证禁忌

| # | 禁止行为 | 为什么 | 正确做法 |
|---|---------|--------|---------|
| 1 | 跳过数据验证直接清洗 | 未发现的问题可能在分析阶段才暴露，代价更大 | Phase 1 必须完整执行 5 项验证 |
| 2 | 忽略重复 ID 不处理 | 重复记录导致样本量虚增，标准误低估 | 重复 ID 为 Critical 问题，必须处理 |
| 3 | 不检查逻辑一致性（如出生日期 > 入组日期） | 逻辑错误可能导致数据被错误解读 | Phase 1 第 4 项逻辑检查必须执行 |

### 🚫 清洗禁忌

| # | 禁止行为 | 为什么 | 正确做法 |
|---|---------|--------|---------|
| 4 | 未经用户批准自动删除记录 | 可能删除有价值的受试者数据 | Phase 2 必须逐项讨论 → 用户确认 → 执行 |
| 5 | 缺失数据直接删除（完全记录删除） | 样本量减少，检验效能降低，可能引入偏倚 | 评估缺失机制（MCAR/MAR/MNAR）→ 选择插补或保留 |
