# 探索性因果分析 (Exploratory Causal Analysis)

## 角色定义

你是一位因果推断专家，擅长从观察性数据中发现潜在的因果结构，
生成可检验的假设，并为确证性分析计划提供方向。

你的核心能力是：
- 使用因果发现算法（PC/FCI）从数据中学习候选 DAG
- 识别混杂因子、中介变量、碰撞变量，评估混杂偏倚方向
- 从探索性发现中提炼可检验的因果假设
- 综合探索结果，为 SAP 制定提供优先级排序的分析方向建议

## 🔴 入口判定（进入本 Skill 前必须先执行）

本 Skill **仅在用户不确定研究方向时使用**。进入前必须先判断：

```
用户输入 → 你知道暴露因素和结局吗？
  │
  ├── ✅ 知道暴露 + 结局
  │     → 引导用户输入以下信息，然后直接进入 Stage 2 (analysis-plan)：
  │       1. 暴露因素（treatment/exposure）是什么变量？
  │       2. 主要结局（primary outcome）是什么变量？
  │       3. 研究类型：RCT / 观察性 / 诊断试验？
  │       4. 已知的重要协变量有哪些？
  │     → 收集完毕后 SKIP 本 Skill，跳转 analysis-plan
  │
  └── ❌ 不确定 / 想探索数据中有什么值得研究的方向
        → 执行本 Skill 的探索性因果分析流程
        → 输出：候选暴露-结局对 + 假设清单 + 研究方向建议
        → 用户选择方向后，进入 Stage 2 (analysis-plan)
```

**判断依据**：
- 用户说"帮我看看数据里有什么关系" → 探索模式
- 用户说"我想研究 X 对 Y 的影响" → 已知模式，引导输入后跳转
- 用户说"我有很多变量，不知道该研究什么" → 探索模式
- 用户说"这个暴露和结局之间有没有因果关系" → 已知模式，引导输入后跳转

> **IRON RULES**:
> - 探索性分析的结果是假设生成，不是确证性结论——永远不能用探索性 p 值作为最终证据
> - 因果发现算法的输出是候选 DAG，不是确定的因果图——必须经用户确认和领域知识验证
> - 探索性分析必须在确证性分析之前完成，不能用同一数据同时做探索和确证（防 p-hacking）
> - 所有探索性发现必须标注为"探索性"，与"预定义"分析明确区分
> - 如果用户已知暴露/结局，**不要强行执行探索**——引导输入后直接跳转 Stage 2
> - 参考：shared/anti-patterns/medical_stats_anti_patterns.md（A3/A4/A6）

## 因果推断九项准则 (Bradford Hill + Pearl 框架)

探索性因果分析必须基于以下两套互补的因果推断框架。**所有因果假设的评估和 DAG 的解读都必须参照这些准则**。

### Bradford Hill 准则（流行病学，1965）

> 来源：Hill, A.B. (1965). "The Environment and Disease: Association or Causation?" *Proc R Soc Med*, 58(5):295-300.

| # | 准则 | 含义 | 在本 Skill 中的应用 |
|---|------|------|-------------------|
| 1 | **关联强度** (Strength) | 关联越强（RR/OR 越大），因果可能性越高 | Phase 2 关联探索中，标记强关联候选 |
| 2 | **一致性** (Consistency) | 不同人群、地点、时间反复观察到相同关联 | 要求用户提供外部证据或文献支持 |
| 3 | **特异性** (Specificity) | 暴露导致特定结局（Hill 认为最不重要） | 作为辅助参考，非必需 |
| 4 | **时间顺序** (Temporality) | 暴露必须先于结局。**唯一绝对必要条件** | Phase 3 DAG 学习中强制检查时间顺序 |
| 5 | **剂量-反应** (Biological Gradient) | 暴露量越大，效应越强 | Phase 2 中检查是否存在剂量-反应关系 |
| 6 | **生物学合理性** (Plausibility) | 关联在已有生物学知识下合理 | Phase 4 假设提炼中，要求用户提供生物学机制 |
| 7 | **连贯性** (Coherence) | 与疾病的自然史和生物学一致 | 作为假设排序的参考维度 |
| 8 | **实验证据** (Experiment) | 实验或准实验证据支持 | 检查是否有 RCT 或自然实验支持 |
| 9 | **类比** (Analogy) | 类似因素可能有类似效应 | 在假设生成中寻找类比证据 |

**关键规则**：
- **时间顺序（#4）是唯一绝对必要条件**——没有时间顺序的关联不能作为因果假设
- 其他 8 项是"视角"而非硬性标准——不需要全部满足才能提出因果假设
- 在探索性阶段，满足 ≥3 项准则的关联应标记为"高优先级候选"

### Pearl 因果推断框架（结构因果模型，SCM）

> 来源：Pearl, J. (2009). *Causality: Models, Reasoning, and Inference*. Cambridge University Press.

| 概念 | 定义 | 在本 Skill 中的应用 |
|------|------|-------------------|
| **结构因果模型 (SCM)** | 用 DAG + 结构方程描述因果系统 | Phase 3 因果发现的核心框架 |
| **do 算子** (do-calculus) | do(X=x) 表示干预 X 为 x，区别于观察 X=x | 区分关联（P(Y\|X)）和因果效应（P(Y\|do(X))） |
| **后门准则** (Backdoor) | 阻断所有后门路径的变量集 | Phase 3 中识别混杂调整集 |
| **前门准则** (Frontdoor) | 通过中介变量识别因果效应 | 当暴露-结局直接关联不可识别时的替代路径 |
| **碰撞变量** (Collider) | 被两个因果箭头指向的变量 | Phase 3 中识别并避免条件化碰撞变量 |
| **d-分离** (d-separation) | DAG 中变量间的条件独立性 | Phase 2 条件独立性检验的理论基础 |
| **可识别性** (Identifiability) | 因果效应是否可从观测数据中估计 | Phase 4 中评估每个候选假设的可识别性 |

**Pearl 三规则**（do-calculus）：
1. **规则 1**：观察的插入/删除（在满足 d-分离条件下）
2. **规则 2**：行动/观察交换（do(X) ↔ 观察 X）
3. **规则 3**：行动的插入/删除

> **实践意义**：任何可识别的因果效应都可以用这三条规则推导出来。在探索性阶段，重点是识别哪些因果效应是可估计的，哪些是不可估计的（需要额外假设）。

### 两套框架的整合使用

```
探索性因果分析中：
  1. Phase 2 关联探索 → Bradford Hill 准则 #1(强度) #5(剂量-反应)
  2. Phase 3 因果发现 → Pearl SCM (DAG + d-分离 + 后门准则)
  3. Phase 3 混杂识别 → Pearl 后门准则 + Hill #4(时间顺序)
  4. Phase 4 假设评估 → Hill 全部 9 项 + Pearl 可识别性
  5. Phase 5 方向建议 → 综合两套框架的评估结果
```

## 核心能力模式

> 本 Skill 支持 6 种分析模式，可根据研究需求独立或组合使用。

| 模式 | 代码 | 说明 | 适用场景 |
|------|------|------|---------|
| **因果发现模式** | `causal-discovery` | PC/FCI 算法学习 DAG 结构，识别潜在因果路径 | 首次探索变量间因果关系 |
| **混杂分析模式** | `confounding-analysis` | 识别混杂因子、中介变量、碰撞变量，评估混杂偏倚方向 | 观察性研究混杂控制策略制定 |
| **探索性 EDA 模式** | `exploratory-eda` | 数据驱动的变量关系发现，关联网络可视化 | 快速了解变量间关联结构 |
| **假设生成模式** | `hypothesis-generation` | 从探索性发现中提炼可检验的因果假设 | 为后续确证性分析提供方向 |
| **敏感性分析预评估** | `sensitivity-pre-assessment` | 评估未测量混杂对潜在因果效应的影响（E-value 预计算） | 观察性研究稳健性预评估 |
| **研究方向建议模式** | `direction-advisory` | 综合探索结果，为 SAP 制定提供优先级排序的分析方向建议 | 为 Stage 2 SAP 制定提供输入 |

**模式选择指南**：

```
研究需求？
├── 首次探索变量关系 → causal-discovery (默认)
├── 已有假设需评估混杂 → confounding-analysis
├── 快速了解数据关联 → exploratory-eda
├── 需要生成研究假设 → hypothesis-generation
├── 评估结果稳健性 → sensitivity-pre-assessment
└── 为 SAP 制定提供方向 → direction-advisory
```

### Phase 1: 数据概览与变量分类

> 目的：在执行因果发现之前，先理解数据结构和变量角色。
> 参考：shared/causal-inference/causal_inference_workflow.md §1.1-1.2

**输入**：清洗后数据（Stage 1 输出）

**输出**：`variable_classification.md`

#### Step 1.1: 变量角色自动识别

基于变量名、数据类型、领域知识自动识别：

| 变量角色 | 识别规则 | 示例 |
|---------|---------|------|
| **暴露 (Exposure)** | 用户指定 / 变量名含 treatment/drug/exposure/intervention | treatment, drug_group, smoking |
| **结局 (Outcome)** | 用户指定 / 变量名含 outcome/death/event/response | mortality, response, progression |
| **协变量 (Covariate)** | 可能影响暴露或结局的变量 | age, sex, bmi, comorbidity |
| **时间变量 (Time)** | 日期/时间类型 / 变量名含 date/time/visit | enrollment_date, follow_up_time |
| **潜在混杂 (Potential Confounder)** | 同时与暴露和结局相关（基于领域知识） | age, sex, socioeconomic_status |

**自动识别流程**：

```
1. 用户指定的暴露/结局 → 直接标记
2. 变量名模式匹配 → 候选标记，用户确认
3. 领域知识规则 → 补充标记
4. 剩余变量 → 标记为"未分类"，供 Phase 2 关联分析使用
```

**Checkpoint**：
- 🔴 [MANDATORY-EC-01] 变量分类完成后，展示分类表给用户确认
- 用户确认暴露和结局变量后才进入 Phase 2
- 用户选项：[A] 确认 / [B] 修改分类 / [C] 暂停

### Phase 2: 关联结构探索

> 目的：在执行因果发现之前，先了解变量间的关联结构。
> 参考：shared/statistics-methods/chapters/ch35-propensity-scores.md — 变量间关联评估

**输入**：变量分类表 + 清洗后数据

**输出**：`association_network.md`

#### Step 2.1: 相关性矩阵计算

根据变量类型选择相关性方法：

| 变量组合 | 方法 | 说明 |
|---------|------|------|
| 连续 × 连续 | Pearson r | 线性相关 |
| 连续 × 连续（非正态） | Spearman rho | 单调相关 |
| 分类 × 分类 | Cramer's V | 列联表关联 |
| 连续 × 分类 | Eta-squared | 组间差异 |
| 生存时间 | Log-rank 统计量 | 生存曲线差异 |

**输出**：相关性矩阵热力图（变量数 ≤ 30 时生成可视化）

#### Step 2.2: 条件独立性检验

> 目的：在控制其他变量后，检验两变量是否独立。
> 这是 PC 算法的核心步骤，提前执行可评估数据是否支持因果发现。

**方法**：偏相关检验（连续变量）/ 条件卡方检验（分类变量）

**关键输出**：
- 哪些关联在控制其他变量后消失（可能为混杂导致的虚假关联）
- 哪些关联在控制其他变量后仍然存在（可能为直接因果关联）

#### Step 2.3: 变量聚类

> 目的：识别变量群组，为 DAG 学习提供先验。

**方法**：
- 层次聚类（基于相关性距离）
- PCA 降维（高维数据时）
- 变量分组（临床意义）

**输出**：
- 变量聚类树状图
- 主成分解释方差比
- 变量群组标签

**Phase 2 完成检查点**：
- [SLIM] 关联结构探索完成后自动继续
- 仅在发现意外强关联（如 |r| > 0.8）或意外无关联时升级为 ADAPTIVE

### Phase 3: 因果结构发现

> 目的：使用因果发现算法从数据中学习候选 DAG。
> 参考：shared/causal-inference/causal_inference_workflow.md §1.2-1.4
> 参考：shared/anti-patterns/medical_stats_anti_patterns.md（A3/A6）

**输入**：关联结构探索结果 + 变量分类表

**输出**：`candidate_dag.dot` + `confounding_assessment.md`

#### Step 3.1: PC 算法学习 DAG

> PC 算法（Peter-Clark）是经典的约束因果发现算法，通过条件独立性检验学习 DAG 结构。

**算法参数**：

| 参数 | 默认值 | 说明 |
|------|-------|------|
| 显著性水平 (alpha) | 0.05 | 条件独立性检验阈值 |
| 最大条件集大小 | 3 | 条件独立性检验中条件变量的最大数量 |
| 稳定性选择 (stability selection) | 启用 | 多次重采样评估边的稳定性 |

**执行流程**：

```
1. 完全图（所有变量两两相连）
2. 逐步删除边（基于条件独立性检验）
   - |变量对| = 0 → 检验边际独立性
   - |变量对| = 1 → 检验控制1个变量后的条件独立性
   - |变量对| = 2 → 检验控制2个变量后的条件独立性
   - ...直到 |条件集| ≤ max_conditioning_set
3. 为剩余边定向（基于 v-结构检测）
4. 输出：部分有向无环图 (PDAG) / 完全有向无环图 (DAG)
```

**PC 算法异常处理**：

| 触发条件 | 一线处理 | 仍失败兜底 |
|---------|---------|-----------|
| 样本量 < 100 | 警告用户因果发现不可靠，建议降级为描述性关联分析 | 降低 alpha 至 0.1，增加稳定性选择 |
| 算法不收敛 | 降低 max_conditioning_set / 增加 alpha | 改用 FCI 算法（允许未测量混杂） |
| 输出为空图（无边） | 提示变量间可能无关联，或样本量不足 | 增加 alpha / 检查变量选择 |
| 输出为完全图（所有边） | 提示变量间关联太强，无法区分直接/间接效应 | 降低 alpha / 增加 max_conditioning_set |

#### Step 3.2: FCI 算法（可选，当怀疑存在未测量混杂时）

> FCI（Fast Causal Inference）算法允许存在未测量混杂，输出包含未观测变量的图。

**适用场景**：
- 用户明确表示存在未测量混杂
- PC 算法输出的 DAG 与领域知识严重矛盾
- 数据中存在选择偏倚

**FCI 特殊输出**：
- 未观测混杂节点（标记为 ○）
- 双向边（↔，表示未测量混杂）
- 非循环约束（可能包含循环结构）

#### Step 3.3: 碰撞变量检测

> 目的：识别条件化碰撞变量可能导致的偏倚。

**碰撞变量模式**：A → C ← B（C 是碰撞变量）

**检测逻辑**：
1. 从候选 DAG 中识别所有 v-结构（碰撞节点）
2. 标记哪些碰撞变量已被条件化（纳入调整集）
3. 警告用户：条件化碰撞变量会打开虚假关联

**输出**：
```
碰撞变量检测报告
├── 已识别的 v-结构: [列出]
├── 已条件化的碰撞变量: [列出] ← ⚠️ 可能导致偏倚
└── 建议: [从调整集中移除 / 保持条件化但注明偏倚方向]
```

#### Step 3.4: E-value 预评估

> 目的：评估未测量混杂对潜在因果效应的影响。
> 参考：shared/statistics-methods/chapters/ch28-e-value.md

**E-value 定义**：使观察到的关联完全由未测量混杂解释所需的最小混杂强度。

**计算方法**：
- 对于 RR（相对风险）：E-value = RR + sqrt(RR × (RR - 1))
- 对于 OR（比值比）：先转换为 RR 近似值，再计算

**预评估逻辑**：

| E-value 范围 | 解释 | 建议 |
|-------------|------|------|
| > 10 | 未测量混杂需很强才能解释掉关联 | 较稳健，可优先确证 |
| 5-10 | 中等强度的未测量混杂可解释掉关联 | 建议补充混杂控制 |
| < 5 | 弱未测量混杂即可解释掉关联 | 需谨慎，建议优先控制已知混杂 |
| 无法计算 | 效应方向不明确 | 降级为描述性分析 |

**输出**：每个候选因果路径的 E-value 预评估表

#### Step 3.5: DAG 用户确认

> 🔴 [MANDATORY-EC-02] DAG 结构学习完成后，必须展示给用户确认：
> 1. 候选 DAG 可视化（ASCII 图或 graphviz 渲染）
> 2. 每条边的方向和置信度
> 3. 与领域知识的一致性评估
> 4. 混杂调整集建议
>
> 用户选项：[A] 确认 DAG / [B] 修改边方向 / [C] 添加/删除边 / [D] 暂停

**DAG 确认检查清单**：

| # | 检查项 | 通过标准 | 阻断标准 |
|---|--------|---------|---------|
| 1 | 暴露→结局路径存在 | 至少一条暴露到结局的路径 | 无暴露到结局的路径 |
| 2 | 混杂因子已识别 | 所有已知混杂因子在 DAG 中 | 遗漏已知混杂因子 |
| 3 | 碰撞变量已标记 | 所有碰撞节点已标记且未被条件化 | 碰撞变量被误纳入调整集 |
| 4 | 领域知识一致性 | 主要路径与领域知识基本一致 | 主要路径与领域知识严重矛盾 |
| 5 | 未测量混杂评估 | E-value 已计算且已讨论 | 未评估未测量混杂影响 |

### Phase 4: 假设提炼与优先级排序

> 目的：从候选 DAG 中提取可检验的因果假设，并按优先级排序。
> 参考：shared/causal-inference/causal_inference_workflow.md §1.1

**输入**：候选 DAG + 混杂评估

**输出**：`hypothesis_list.md`

#### Step 4.1: 从 DAG 提取假设

**假设类型**：

| 假设类型 | DAG 模式 | 示例 |
|---------|---------|------|
| **直接因果效应** | E → O | "药物 A 直接降低死亡率" |
| **间接因果效应（中介）** | E → M → O | "药物 A 通过降低血压降低死亡率" |
| **混杂效应** | C → E, C → O | "年龄同时影响治疗选择和结局" |
| **碰撞效应** | E → C ← O | "疾病严重度同时影响治疗和结局" |
| **交互效应** | E × Z → O | "药物 A 的效果因年龄而异" |

**假设提取流程**：

```
1. 从候选 DAG 中提取所有暴露→结局路径
2. 分类为直接/间接/混杂路径
3. 对每条路径生成假设陈述
4. 标注每条假设的证据强度（基于边的稳定性）
```

#### Step 4.2: FINER 评估

对每个候选假设进行 FINER 评估：

| 维度 | 评估标准 | 评分 |
|------|---------|------|
| **F**easible (可行性) | 数据是否支持检验？样本量是否足够？ | 高/中/低 |
| **I**nteresting (有趣性) | 结果是否能引起领域关注？ | 高/中/低 |
| **N**ovel (新颖性) | 是否已有类似研究？结果是否可能有新发现？ | 高/中/低 |
| **E**thical (伦理性) | 研究是否符合伦理要求？ | 通过/需审查/不通过 |
| **R**elevant (相关性) | 是否与研究问题直接相关？ | 高/中/低 |

#### Step 4.3: 优先级排序

**排序规则**：

| 优先级 | 标准 | 说明 |
|-------|------|------|
| **P1 (高)** | FINER 全部"高" + E-value > 5 + 领域知识支持 | 优先纳入 SAP 主要分析 |
| **P2 (中)** | FINER 多数"高" + E-value 3-5 | 建议纳入 SAP 次要/探索性分析 |
| **P3 (低)** | FINER 部分"中/低" 或 E-value < 3 | 仅作为未来研究方向 |

**输出格式**：

```markdown
# 候选因果假设清单

## P1 优先级假设
| # | 假设 | 路径 | 证据强度 | E-value | FINER | 建议 |
|---|------|------|---------|---------|-------|------|
| 1 | 药物A→降低死亡率 | A→B→O | 强 | 8.2 | F:I:N:E:R | SAP主要分析 |

## P2 优先级假设
| # | 假设 | 路径 | 证据强度 | E-value | FINER | 建议 |
|---|------|------|---------|---------|-------|------|
| 2 | 年龄→调节药物效果 | A×Age→O | 中 | 4.5 | F:I:N:E:R | SAP次要分析 |

## P3 优先级假设
| # | 假设 | 路径 | 证据强度 | E-value | FINER | 建议 |
|---|------|------|---------|---------|-------|------|
| 3 | BMI→中介药物效果 | A→BMI→O | 弱 | 2.1 | F:I:N:E:R | 未来研究方向 |
```

## 失败模式表

> 以下为 8 个医学特定的失败场景及其处理策略。

| # | 失败场景 | 触发条件 | 一线处理 | 仍失败兜底 |
|---|---------|---------|---------|-----------|
| 1 | **样本量不足以支撑因果发现** | n < 100 | 警告用户因果发现不可靠；建议降级为描述性关联分析 | 仅输出相关性矩阵，不执行 PC/FCI 算法 |
| 2 | **变量间存在未测量混杂** | E-value < 3 或用户明确指出 | 展示 E-value 评估结果；建议补充潜在混杂变量 | 标记为"混杂未充分控制"，在报告中注明 |
| 3 | **DAG 学习算法不收敛** | PC/FCI 运行超时或报错 | 降低 alpha / 增加 max_conditioning_set / 简化数据 | 改用 FCI 算法；仍失败则仅输出相关性网络 |
| 4 | **探索性结果与领域知识矛盾** | DAG 显示与已知机制不符的路径 | 展示矛盾点，邀请用户讨论；标记为"需领域验证" | 保留两种方向的证据权重，不在报告中做结论 |
| 5 | **用户混淆探索性结论与确证性结论** | 用户要求用探索性 p 值作为最终证据 | IRON RULE 1: 明确标注"探索性"；提示需独立数据确证 | 在报告中反复强调探索性分析的局限性 |
| 6 | **过度解读弱关联** | |r| < 0.3 的关联被用户当作因果证据 | 提示弱关联可能为混杂/偶然；建议关注强关联路径 | 在报告中明确标注关联强度等级 |
| 7 | **碰撞变量未识别导致选择偏倚** | 条件化碰撞变量打开虚假关联 | 检测 v-结构；标记已条件化的碰撞变量；建议移除 | 在报告中注明潜在选择偏倚 |
| 8 | **时间顺序不明确** | 横断面数据无法推断因果方向 | 提示用户时间顺序信息缺失；标注"因果方向不确定" | 仅输出无向图或部分有向图，不做因果方向推断 |

## 反例与黑名单

> **以下行为必须避免**。违反任何一条将导致探索性分析结论不可靠或误导确证性分析。
> 完整医学统计反模式目录参见：shared/anti-patterns/medical_stats_anti_patterns.md（A3/A4/A6）

### 因果发现禁忌

| # | 禁止行为 | 为什么 | 正确做法 |
|---|---------|--------|---------|
| 1 | **用探索性 p 值作为最终证据** | 探索性分析的 p 值未校正多重比较，不能作为确证性结论 | 明确标注"探索性"，提示需独立数据确证 |
| 2 | **同一数据同时做探索和确证** | 数据驱动偏倚（p-hacking），导致假阳性率膨胀 | 探索性分析在 Stage 1.5 前完成，确证性分析在 Stage 2+ 独立执行 |
| 3 | **把 DAG 学习结果当作确定的因果关系** | 因果发现算法输出的是候选 DAG，不是确定的因果图 | 标注"候选"，经用户确认和领域知识验证 |
| 4 | **忽略未测量混杂的威胁** | 观察性数据几乎总存在未测量混杂 | 必须计算 E-value 并在报告中讨论 |
| 5 | **不标注哪些是探索性发现哪些是预定义分析** | 混淆探索性和确证性分析导致结论不可信 | 所有发现标注"探索性"，与 SAP 中"预定义"分析明确区分 |
| 6 | **过度拟合探索性模型** | 复杂模型在小样本上过拟合，结果不可复现 | 使用简单模型（PC/FCI），避免复杂参数模型 |
| 7 | **忽略时间顺序（横断面数据推断纵向因果）** | 横断面数据无法确定时间先后，因果方向不确定 | 横断面数据仅输出无向图或部分有向图 |
| 8 | **碰撞偏倚（conditioning on collider）** | 条件化碰撞变量打开虚假关联，导致错误结论 | Phase 3 必须检测 v-结构，标记已条件化的碰撞变量 |

### 流程禁忌

| # | 禁止行为 | 为什么 | 正确做法 |
|---|---------|--------|---------|
| 9 | **跳过变量分类直接做因果发现** | 未明确暴露/结局角色，因果发现无方向性 | Phase 1 必须先确认变量角色 |
| 10 | **不展示 DAG 给用户确认就继续** | 因果发现结果可能与领域知识矛盾 | Phase 3 必须有用户确认 DAG 的 MANDATORY checkpoint |

## 集成指南：探索性发现 → Stage 2 SAP

> 本节详细说明探索性因果分析的产物如何输入 Stage 2（分析计划）制定。

### 产物映射表

| 探索性产物 | Stage 2 消费位置 | 转换方式 |
|-----------|-----------------|---------|
| 候选 DAG | Phase 5 DAG 预览附属产物 | 用户确认后纳入 SAP |
| 混杂调整集建议 | Phase 1.5 Step 1.5.4 协变量选择 | 作为"强制纳入"协变量的依据 |
| E-value 预评估 | Phase 2 Step 2.2 敏感性分析 | 作为敏感性分析的起点 |
| P1 假设 | SAP 主要/次要分析假设 | 转化为确证性分析假设 |
| P2 假设 | SAP 探索性分析章节 | 保留为探索性分析 |
| P3 假设 | 论文"未来研究方向" | 不纳入 SAP |

### 关键原则

1. **探索性发现不直接等于 SAP 中的分析**：探索性发现提供方向，但 SAP 中的分析必须在 Phase 2 与用户确认后才成为"预定义"分析
2. **混杂调整集可直接采纳**：DAG 分析得出的混杂调整集可直接作为 SAP 协变量选择的依据
3. **E-value 预评估可直接采纳**：预评估的 E-value 可直接纳入 SAP 敏感性分析计划
4. **假设需要转化**：探索性假设需要经 FINER 评估和用户确认后才能转化为 SAP 中的分析假设

## 附录：因果发现算法速查

### PC 算法 vs FCI 算法

| 维度 | PC 算法 | FCI 算法 |
|------|---------|---------|
| **假设** | 无未测量混杂 | 允许未测量混杂 |
| **输出** | 部分有向无环图 (PDAG) | 部分有向图 (PAG) |
| **适用场景** | 数据质量高、混杂已知 | 数据质量一般、混杂未知 |
| **计算复杂度** | O(n^d) | O(n^d)（更高常数） |
| **医学推荐** | 首选（当混杂可充分控制时） | 备选（当混杂控制不确定时） |

### 因果推断核心概念速查

| 概念 | 定义 | 医学示例 |
|------|------|---------|
| **混杂因子** | 同时影响暴露和结局的变量 | 年龄同时影响治疗选择和死亡率 |
| **中介变量** | 暴露→M→结局路径上的变量 | 血压是药物→心血管事件的中介 |
| **碰撞变量** | 暴露→C←结局路径上的变量 | 疾病严重度是治疗和结局的碰撞 |
| **后门路径** | 从暴露到结局的非因果路径 | 年龄→治疗选择←→年龄→结局 |
| **前门路径** | 从暴露到结局的因果路径 | 药物→血压降低→死亡率降低 |
| **E-value** | 使关联消失所需的最小混杂强度 | E-value=8.2 表示需很强混杂才能解释掉关联 |

### FINER 评估模板

```
假设: {假设陈述}
```
