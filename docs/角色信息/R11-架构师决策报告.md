## 二、技术栈选型

### FHIR模块 (#1)
- **核心库**: `fhir.resources` (Pydantic v2 FHIR R4/R4B模型)
- **HTTP客户端**: `httpx` (异步FHIR服务器通信)
- **测试服务器**: HAPI FHIR JPA Server (公开测试实例)
- **参考**: Smart-on-FHIR/client-py, beda-software/fhirpy

### OMOP模块 (#2)
- **数据读取**: `pandas` + `sqlalchemy` (CSV/数据库双模式)
- **概念映射**: 本地OMOP Vocabulary CSV + Athena下载
- **轻量查询**: DuckDB (OMOP-on-DuckDB模式，无需数据库服务器)
- **参考**: OHDSI/CommonDataModel, EHDEN/omop-etl-py

### 术语编码 (#3)
- **ICD-10**: `icd10-cm` (本地字典) + WHO ICD API
- **SNOMED CT**: SNOMED CT International API + 本地子集
- **LOINC**: `loinc` Python包 + 本地CSV
- **模糊匹配**: `rapidfuzz` (编码建议)

### CDISC (#4)
- **核心**: 自实现(SDTM/ADaM映射逻辑从R模板移植)
- **XPT读取**: `pyreadstat` (SAS Transport格式)
- **验证**: 自定义SDTM/ADaM验证规则引擎

## 五、风险与缓解

| 风险 | 概率 | 影响 | 缓解措施 |
|------|------|------|---------|
| FHIR规范复杂度超预期 | 中 | 高 | Phase 1仅支持Patient+Observation，渐进扩展 |
| OMOP概念映射不完整 | 中 | 中 | 使用Athena标准下载，覆盖Top 1000概念 |
| 术语API限流 | 低 | 中 | 本地缓存+批量查询+离线模式 |
| R→Python移植逻辑丢失 | 低 | 高 | 逐函数对照测试，保持输出一致性 |
| 新模块与现有架构冲突 | 低 | 中 | 遵循现有msra_modules/和shared/约定 |
