# 数据契约规范 — Cross-Stage Data Contracts

> **文件**: `shared/handoff_schemas.md` | **版本**: v1.0

---

## 1. 概述

数据契约定义了临床研究流水线各阶段之间传递的每一个 artifact 的精确数据结构。所有产生或消费这些 artifact 的 agent 必须严格遵守 schema 定义。

### 1.1 设计原则

| 原则 | 说明 |
|------|------|
| **Schema 强制** | 所有跨阶段交接必须符合预定义 Schema |
| **必填验证** | 消费方在接收时验证所有必填字段 |
| **失败路径** | 缺失必填字段触发 `HANDOFF_INCOMPLETE` |
| **Markdown 格式** | 所有 schema 采用 Markdown 结构化输出 |

### 2.1 Schema 1: PICO/PICOS 研究问题定义

**生产者**: `data-prep` (Stage 1) / `analysis-plan` (Stage 2)
**消费者**: `analysis-plan` (Stage 2) / `report` (Stage 4/5)

|------|------|------|
| `research_question` | string | PICO/PICOS 框架结构化的研究问题 |
| `pico` | object | PICO/PICOS 各要素 |

| `target_population` | object | 目标人群描述 + 纳排标准 |
| `primary_endpoint` | string | 主要终点指标及定义 |
| `secondary_endpoints` | list[string] | 次要终点列表 |
| `hypothesis` | object | 主要/次要假设 |

| 字段 | 类型 | 说明 |

## 3. 交接验证规则

### 3.1 验证流程

```
Agent A 完成工作
    │
    ▼
生成 Handoff (符合 Schema)
    │
    ▼
Agent B 接收
    │
    ├─ 验证必填字段
    │   ├─ 全部存在 → 继续
    │   └─ 缺失 → HANDOFF_INCOMPLETE → 要求重新生成
    │
    ├─ 验证字段类型
    │   ├─ 类型正确 → 继续
    │   └─ 类型错误 → SCHEMA_VIOLATION → 要求修正
    │
    └─ 验证业务规则
        ├─ 规则满足 → 接受
        └─ 规则违反 → BUSINESS_RULE_VIOLATION → 上报
```

### 3.2 错误处理

| 错误码 | 说明 | 处理方式 |
|--------|------|---------|
| `HANDOFF_INCOMPLETE` | 缺失必填字段 | 要求生产者重新生成 |
| `SCHEMA_VIOLATION` | 字段类型错误 | 要求生产者修正 |
| `BUSINESS_RULE_VIOLATION` | 业务规则违反 | 上报 ConflictResolver |
| `VERSION_MISMATCH` | Schema 版本不匹配 | 要求升级/降级 |

## 5. 子 Agent Handoff Schema

混合模式（模式 C）下，子 Agent 间通过文件交接。除了上述 12 个 Schema 外，子 Agent 交接还需遵循以下规范：

### 5.1 子 Agent 启动 Handoff

Orchestrator 启动子 Agent 时，在 prompt 中包含标准化输入：

```markdown
## Agent 任务指令

### 角色定义
你是 [Agent 角色描述]。

### 输入产物
- [产物1]: [文件路径]
- [产物2]: [文件路径]

### 验证标准
- [检查项1]: [通过标准]
- [检查项2]: [通过标准]

### 期望输出格式
请按以下 Handoff 格式输出结果...

### 约束
- 不得修改输入文件
- 不得访问其他阶段的产物
- 输出必须写入指定路径
```

### 5.2 子 Agent 输出 Handoff

子 Agent 完成后，输出标准化 Handoff 报告：

```markdown
## Handoff: [Agent Name]

### 已完成工作
- [任务1]: [结果]
- [任务2]: [结果]

### 产物路径
- [产物1]: [文件路径]
- [产物2]: [文件路径]

### 验证方法
- [如何验证产物正确性]

### 已知问题
- [问题描述]

### 待决策项
- [需要 Orchestrator/用户决定的事项]
```

### 5.3 Orchestrator 验证流程

```
收到子 Agent 输出
    │
    ├─ 检查 Handoff 格式完整性
    │    ├─ 缺失字段 → HANDOFF_INCOMPLETE → 要求子 Agent 修正
    │    └─ 完整 → 继续
    │
    ├─ 验证产物文件存在性
    │    ├─ 文件缺失 → 要求子 Agent 重新生成
    │    └─ 文件存在 → 继续
    │
    ├─ 验证 Schema 合规性
    │    ├─ 违反 → SCHEMA_VIOLATION → 要求修正
    │    └─ 合规 → 接受
    │
    └─ 记录到 Material Passport
```
