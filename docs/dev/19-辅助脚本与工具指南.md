# 19. 辅助脚本与工具指南

## 19.1 概述

MSRA 项目在 `scripts/` 目录下维护了 11 个辅助脚本，这些脚本为开发、测试、部署和维护提供了自动化工具支持。它们不属于核心 Pipeline，但对于保证代码质量、验证系统完整性和简化运维流程至关重要。

### 19.1.1 脚本分类

| 类别 | 脚本 | 用途 |
|------|------|------|
| **测试与验证** | benchmark_large_scale.py | 大规模处理引擎性能基准测试 |
| | check_pipeline_integrity.py | Pipeline 完整性检查 |
| | check_sprint_contract.py | Sprint 合约验证 |
| | eval_pipeline_gold.py | Pipeline 黄金标准评估 |
| **代码质量** | lint_gate_counts.py | 门闸检查项数量 lint |
| | lint_skill_structure.py | Skill 结构规范 lint |
| | migrate_print_to_logging.py | print 语句迁移到 logging |
| **部署与运维** | generate_msra_handoff_bundle.py | 生成 Handoff Bundle |
| | auto_push_optimization.sh | 自动推送优化 |
| | check_optimization_status.py | 优化状态检查 |

## 19.2 各脚本功能说明和使用方式

### 19.2.1 benchmark_large_scale.py — 大规模处理引擎基准测试

**功能说明**

对大规模数据处理引擎（Polars、Dask、DuckDB）进行性能基准测试，帮助用户选择最适合其数据规模的引擎。

**使用方式**

```bash
# 运行完整基准测试
python scripts/benchmark_large_scale.py

# 指定数据规模
python scripts/benchmark_large_scale.py --rows 1000000 --cols 100

# 测试特定引擎
python scripts/benchmark_large_scale.py --engines polars,dask

# 输出详细报告
python scripts/benchmark_large_scale.py --verbose --output benchmark_report.json
```

**输出内容**

- 各引擎的执行时间对比
- 内存使用峰值
- CPU 利用率
- 推荐引擎建议（基于数据规模）

**典型使用场景**

- 数据量超过 100MB 时选择合适的处理引擎
- 性能调优时对比不同引擎的瓶颈
- 系统升级后验证性能回归

### 19.2.2 check_pipeline_integrity.py — Pipeline 完整性检查

**功能说明**

验证 MSRA Pipeline 的完整性，检查所有必需文件、依赖关系和配置是否正确。

**使用方式**

```bash
# 运行完整检查
python scripts/check_pipeline_integrity.py

# 检查特定阶段
python scripts/check_pipeline_integrity.py --stage 1

# 详细输出
python scripts/check_pipeline_integrity.py --verbose

# 检查特定项目目录
python scripts/check_pipeline_integrity.py --project /path/to/project
```

**检查项目**

1. **文件完整性**: 所有必需的配置文件、模板文件是否存在
2. **依赖完整性**: Python 包、R 包、外部工具是否安装
3. **Schema 完整性**: JSON Schema 文件是否有效
4. **配置完整性**: 必需配置项是否设置
5. **目录结构**: 输出目录结构是否正确

**输出示例**

```
Pipeline Integrity Check
========================
✅ File integrity: 45/45 files present
✅ Dependency integrity: 12/12 packages installed
⚠️ Schema integrity: 1 warning (evaluator/full.json uses draft-07)
✅ Config integrity: all required keys set
✅ Directory structure: valid

Summary: 1 warning, 0 errors
```

### 19.2.3 check_sprint_contract.py — Sprint 合约检查

**功能说明**

验证 Sprint 合约 JSON 文件是否符合 Schema 13.1 规范。

**使用方式**

```bash
# 验证单个合约
python scripts/check_sprint_contract.py shared/contracts/reviewer/full.json

# 验证多个合约
python scripts/check_sprint_contract.py shared/contracts/reviewer/*.json

# 指定 MSRA 版本
python scripts/check_sprint_contract.py contract.json --MSRA-version v3.6.2
```

**验证规则**

1. **必需字段**: `panel_size`, `acceptance_dimensions`, `failure_conditions`
2. **severity 枚举**: 必须为 `critical`, `major`, `minor`, `info` 之一
3. **panel_size**: 必须为正整数
4. **failure_conditions**: 必须包含 severity 字段

**输出示例**

```
✅ shared/contracts/reviewer/full.json: OK
✅ shared/contracts/reviewer/methodology_focus.json: OK
❌ shared/contracts/invalid.json: FAIL
   - Missing required field: panel_size
   - failure_conditions[0]: missing 'severity'
```

### 19.2.4 eval_pipeline_gold.py — Pipeline 评估（对照黄金标准）

**功能说明**

使用预定义的黄金标准数据集 (Gold Tuples) 评估 Pipeline 的检测能力，验证数据质量检查的准确性。

**使用方式**

```bash
# 运行完整评估
PYTHONPATH=. python -m scripts.eval_pipeline_gold

# 评估特定 tuple
PYTHONPATH=. python -m scripts.eval_pipeline_gold --tuple 001

# 详细输出
PYTHONPATH=. python -m scripts.eval_pipeline_gold --verbose

# 输出评估报告
PYTHONPATH=. python -m scripts.eval_pipeline_gold --output eval_report.json
```

**黄金标准数据集 (Gold Tuples)**

| Tuple ID | 检测目标 | 预期结果 |
|----------|----------|----------|
| 001 | 负值 BillingAmount | 应检出负值记录 |
| 002 | 异常 Age（负值或>120） | 应检出异常年龄 |
| 003 | Age 缺失 | 应检出缺失值 |
| 004 | 重复 PatientID | 应检出重复记录 |
| 005 | 性别-诊断逻辑矛盾 | 应检出男+卵巢囊肿/女+前列腺癌 |
| 006 | 日期逻辑错误 | 应检出出院<入院或未来日期 |
| 007 | 不可能生命体征 | 应检出心率=0/>250, 体温<35/>42 |
| ... | ... | ... |

**输出示例**

```
Pipeline Gold Evaluation
========================
Test data: tests/msra_test_data.csv
Tuples evaluated: 21/21

Results:
  ✅ 001-negative-billing-amount: PASS (detected: true, count: 3)
  ✅ 002-abnormal-age: PASS (detected: true, count: 2)
  ✅ 003-missing-age: PASS (detected: true, count: 5)
  ...

Score: 20/21 (95.2%)
```

### 19.2.5 generate_msra_handoff_bundle.py — 生成 Handoff Bundle

**功能说明**

生成 Stage 5.0 Paper Intake 所需的 MSRA Handoff Bundle 文件，将统计分析产物传递给论文写作模块。

**使用方式**

```bash
# 基本用法
python scripts/generate_msra_handoff_bundle.py \
    --passport MSRA/passport.json \
    --sap MSRA/SAP.md \
    --output MSRA/handoff_bundle.json

# 在代码中调用
from scripts.generate_msra_handoff_bundle import generate_handoff_bundle

pm = PassportManager("path/to/passport.json")
bundle_path = generate_handoff_bundle(pm, sap_path="MSRA/SAP.md")
```

**Bundle 内容**

```json
{
  "bundle_version": "1.0",
  "generated_at": "2026-06-23T10:00:00Z",
  "research_question": "研究问题描述",
  "methods_summary": "统计方法摘要",
  "tables": ["tables/Table1.docx", "tables/Table2.docx"],
  "figures": ["figures/Figure1.png", "figures/Figure2.png"],
  "sap_path": "MSRA/SAP.md",
  "passport_summary": {
    "data_preparation": {...},
    "analysis_execution": {...}
  }
}
```

**使用场景**

- 完成统计分析后，准备数据传递给论文写作模块
- 交接统计分析产物给其他团队成员
- 归档分析结果

### 19.2.6 lint_gate_counts.py — 门闸检查项数量 lint

**功能说明**

检查所有 Gate 的检查项数量是否符合规范，确保 Gate 定义的完整性。

**使用方式**

```bash
# 运行 lint 检查
python scripts/lint_gate_counts.py

# 检查特定 Gate
python scripts/lint_gate_counts.py --gate DG-01

# 输出详细信息
python scripts/lint_gate_counts.py --verbose
```

**检查规则**

1. 每个 Gate 必须至少有 3 个检查项
2. 检查项必须有明确的通过/失败标准
3. 检查项编号必须连续
4. 不允许重复的检查项描述

**输出示例**

```
Gate Lint Results
=================
✅ DG-01 (数据完整性检查): 5 items
✅ DG-02 (变量范围检查): 4 items
⚠️ DG-03 (缺失模式检查): 2 items (minimum: 3)
✅ DG-04 (分析人群一致性): 3 items

Summary: 1 warning, 0 errors
```

### 19.2.7 migrate_print_to_logging.py — print 迁移到 logging

**功能说明**

自动将代码中的 `print()` 语句迁移到 `logging` 模块，提高日志管理的专业性。

**使用方式**

```bash
# 预览迁移（不实际修改）
python scripts/migrate_print_to_logging.py --dry-run

# 迁移单个文件
python scripts/migrate_print_to_logging.py path/to/file.py

# 迁移目录
python scripts/migrate_print_to_logging.py --directory scripts/

# 保留原始 print（仅添加 logging）
python scripts/migrate_print_to_logging.py --keep-print

# 指定日志级别映射
python scripts/migrate_print_to_logging.py --level-map '{"debug": "DEBUG", "info": "INFO"}'
```

**迁移规则**

| 原始模式 | 迁移后 |
|----------|--------|
| `print("message")` | `logger.info("message")` |
| `print("Error:", e)` | `logger.error("Error: %s", e)` |
| `print(f"Debug: {var}")` | `logger.debug("Debug: %s", var)` |
| `print("Warning...", file=sys.stderr)` | `logger.warning("Warning...")` |

**输出示例**

```
Migration Preview
=================
File: scripts/example.py
  Line 15: print("Starting...") → logger.info("Starting...")
  Line 23: print(f"Count: {n}") → logger.info("Count: %s", n)
  Line 45: print("Error!", file=sys.stderr) → logger.error("Error!")

Total: 3 changes in 1 file
```

**功能说明**

验证 Skill 目录结构是否符合规范，确保 Skill 的可发现性和可维护性。

```bash
# 检查特定 Skill
python scripts/lint_skill_structure.py --skill report
```

**检查规则**

1. **必需文件**: SKILL.md、__init__.py

### 19.2.9 auto_push_optimization.sh — 自动推送优化

**功能说明**

自动化推送优化结果到远程仓库，包括代码变更、测试结果和性能报告。

**使用方式**

```bash
# 运行自动推送
./scripts/auto_push_optimization.sh

# 指定分支
./scripts/auto_push_optimization.sh --branch feature/optimization

# 跳过测试
./scripts/auto_push_optimization.sh --skip-tests

# 干运行（预览操作）
./scripts/auto_push_optimization.sh --dry-run
```

**执行流程**

1. 检查工作目录状态
2. 运行优化脚本
3. 执行测试套件
4. 生成性能报告
5. 提交变更
6. 推送到远程仓库

**输出示例**

```
Auto Push Optimization
======================
[1/6] Checking working directory... ✅
[2/6] Running optimization... ✅ (3 files changed)
[3/6] Running tests... ✅ (142 passed, 0 failed)
[4/6] Generating performance report... ✅
[5/6] Committing changes... ✅ (commit: abc1234)
[6/6] Pushing to origin/feature/optimization... ✅

Done! Optimization pushed successfully.
```
