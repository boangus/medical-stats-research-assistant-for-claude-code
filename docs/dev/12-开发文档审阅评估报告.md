# 开发文档审阅评估报告

> **审阅日期**: 2026-06-23 | **版本**: v1.0.0 | **审阅者**: 博安 + Engineering Workflow Coach | **状态**: 建议已全部执行

---

## 1. 审阅概述

### 1.1 审阅范围

`docs/dev/` 目录下 12 份文档（00-11），对照实际项目结构进行全面审查。

### 1.2 总体评价

| 维度 | 评分 | 说明 |
|------|------|------|
| **覆盖度** | ★★★★☆ | Pipeline 核心阶段覆盖完整，但有 6 个重要模块/子系统完全缺失 |
| **准确性** | ★★★★☆ | 已有文档描述与实际代码基本一致，个别引用路径需验证 |
| **可操作性** | ★★★☆☆ | 偏重"是什么"，缺少"怎么做"——开发流程文档不足 |
| **可维护性** | ★★★☆☆ | 缺少版本管理、ADR 决策记录、变更日志绑定等维护机制 |

**总结**: 核心流水线的 12 份文档质量不错，对理解系统架构和各模块职责很有帮助。但作为一个完整的开发文档体系，还缺少 **运维指南、贡献者工作流、ADR 决策记录、评估体系文档、以及若干重要子系统的详细设计**。

## 2. 已有文档逐份审阅

### 2.0 项目总览与架构 (`00-项目总览与架构.md`)

**优点**: 架构图清晰（4层分层），Pipeline 阶段表完整，文档索引表实用。

**问题**:
1. **Agent 层显示 6 个 Agent（含 Arbitrator），但文档 §4.1 只列 5 个**。Arbitrator 在架构图中出现却无详细说明，容易造成困惑。
2. **目录结构列出了 `msra_modules/` 的 4 个子模块**（medical_imaging, bioinformatics, realtime_analytics, cross_domain），但没有任何文档描述这些实验性模块的设计和使用。
3. **技术栈部分**仅列出 R/Python 的核心包，未提及实际项目中的大规模处理引擎（Polars/DuckDB/Dask 的选择策略、engine_selector 的自动切换逻辑）。

### 2.1 数据准备模块 (`01-数据准备模块.md`)

**优点**: Iron Rules 明确，流程图覆盖 Phase 0.5-7，Gate 1.5 检查清单详尽。

**问题**:
1. **Phase 编号跳跃**: 流程从 Phase 5 直接到 Phase 7，Phase 6 去向不明。文档未说明是否为设计遗留。
2. **值规范化模块**提及 TCM 术语 GB/T 16751 标准，但未说明 `shared/variable_normalization/` 的具体实现细节和扩展方式。
3. **多数据集模式**仅提了三种（单文件/多文件/多时间点），缺少对 Parquet、数据库连接等实际场景的处理说明。

### 2.2 探索性因果分析模块 (`02-探索性因果分析模块.md`)

**优点**: 入口判定逻辑清晰（已知 vs 未知方向），Bradford Hill + Pearl SCM 双框架完整。

**问题**:
1. **与 `systematic-survey` Skill 的衔接缺失**: 文档未说明探索结果如何触发文献调研，与 11 号文档的差距分析不一致。
2. **FCI 算法实现未说明**: 提到使用 PC 和 FCI 算法，但未说明是调用外部库（如 pgmpy/causal-learn）还是自实现。
3. **模板文件引用路径**: 引用了 `shared/templates/causal/dag_variable_selection_template.R` 等文件，但实际 `shared/causal-inference/` 目录也有因果推断模块，两者关系未说明。

### 2.3 分析计划模块 (`03-分析计划模块.md`)

**优点**: 研究类型分支表（RCT/观察性/诊断试验）非常实用，Estimands 五要素示例清晰。

**问题**:
1. **Quick 模式未说明触发条件**: 何时推荐 Quick 模式 vs 标准模式？缺少决策指引。
2. **SAP 模板结构与实际模板文件的对应关系**需验证——文档引用了 `shared/sap/templates/` 但实际路径可能不同。
3. **样本量计算**仅简略提及，`shared/sample-size/` 子目录的详细设计未在文档中体现。

### 2.4 分析执行模块 (`04-分析执行模块.md`)

**优点**: Generator-Evaluator 模式设计精良，自愈 Debug 五级机制实用，变更管理流程完整。

**问题**:
1. **13 项质量清单**在 Phase 4 提及但未列出具体内容——应展开。
2. **代码可复现性**仅提"随机种子已设置"，实际的 `shared/reproducibility/` 模块有完整的审计器和报告生成器，文档未覆盖。
3. **错误诊断系统**未提及: `shared/error_diagnostics/` 包含自动修复建议（auto_fix_suggestions.py/R）和错误模式库（error_patterns.md），但文档中完全未出现。

### 2.5 报告生成模块 (`05-报告生成模块.md`)

**优点**: 双模式设计（报告 vs 论文）清晰，报告规范支持列表全面（16 个），期刊模板列表完整。

**问题**:

### 2.6 论文写作模块 (`06-论文写作模块.md`)

3. **`shared/references/` 中的实用工具**（protected_hedging_phrases.md, word_count_conventions.md）仅简略提及，缺少使用示例。

### 2.9 数据契约规范 (`09-数据契约规范.md`)

**问题**:

2. **JSON Schema 文件未提及**: `shared/contracts/` 下有多个 JSON Schema 文件（sprint_contract.schema.json, passport schemas, audit schemas, evaluator schemas, reviewer schemas），文档仅提了 sprint_contract 和 passport，其余未覆盖。

### 2.10 代码模板库 (`10-代码模板库.md`)

**优点**: 模板分类清晰，覆盖领域广泛。

**问题**:
1. **缺少以下模板分类的说明**:

   - `shared/templates/survival/` — 生存分析模板
   - `shared/templates/regression/` — 回归分析模板

3. **R 模板与 Python 模板的比例**未说明——实际项目中 R 模板较少（主要在 causal/），Python 模板占多数。

### 2.11 现状评估与差距分析 (`11-现状评估与差距分析.md`)

**优点**: 逐项对比用户期望 vs 当前实现，路线图分短/中/长期。

**问题**:
1. **完成度 97% 偏高**: 文档自身列出的差距（文献调研未深度集成、测试覆盖不足、Agent 框架集成度问题）与 97% 的评分不完全匹配。建议降低到 92-95% 更诚实。
2. **差距分析遗漏了多个实际存在的模块**: calibration、error_diagnostics、reproducibility、large_scale_processing 等模块在差距分析中完全未提及。
3. **长期路线图过于简略**: "Web UI 界面" 和 "多语言支持" 缺少可行性分析。
