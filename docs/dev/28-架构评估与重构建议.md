# MSRA 架构评估与重构建议

> **版本**: v1.0.0 | **评估日期**: 2026-06-24 | **评估范围**: 全模块架构合规性、模块化质量、多Agent编排适用性

---

## 一、评估总览

### 1.1 评估维度与评分

| 维度 | 评分(1-10) | 说明 |
|------|-----------|------|
| **架构设计** | 9 | 分层清晰，Pipeline+Skill+Agent三层设计精良 |
| **模块化质量** | 7 | 边界基本清晰，存在3处可优化的冗余/耦合 |
| **文档合规性** | 8.5 | 28份dev文档完备，存在少量文档-代码不一致 |
| **代码实现深度** | 6 | Python代码质量高，但大部分逻辑由LLM执行而非代码 |
| **测试覆盖** | 7.5 | 511个测试通过，Pipeline集成测试可增强 |
| **多Agent适用性** | 8.5 | 混合模式(模式C)设计合理，已有完整的降级策略 |
| **总体评价** | **7.7/10** | 设计精良的医学统计Pipeline，核心优势在知识库深度而非代码复杂度 |

### 1.2 一句话结论

MSRA 的核心竞争力在于 **丰富的医学统计知识库 + 严格的Pipeline质量门闸设计**，而非代码实现本身。它更像一个"高度结构化的LLM工作流规范"，而非传统的软件系统。这种定位是合理的——但需要明确承认并据此制定演进策略。

## 四、多Agent编排评估

### 4.1 适用性分析

MSRA 的多Agent编排需求分析：

| 维度 | 评估 | 说明 |
|------|------|------|
| **任务独立性** | 🟢 高 | 各Stage有明确输入输出，可独立执行 |
| **上下文隔离需求** | 🟢 高 | Exec Inference需要独立于Exec Runner，QC需要独立于生成过程 |
| **并行执行机会** | 🟡 中 | Gate检查可与下一阶段准备并行，但大部分Stage有强依赖 |
| **冲突频率** | 🟢 低 | 方法选择、数据清洗等决策点有明确的人机回环机制 |
| **降级容忍度** | 🟢 高 | 已设计完整的降级策略，功能不受影响 |

**结论：适合引入多Agent编排**，特别是混合模式(模式C)。

### 4.2 推荐的Agent拆分方案

基于当前架构，推荐以下Agent分配：

```
                    ┌──────────────────────────┐
                    │   Pipeline Orchestrator   │
                    │   (主对话, Skill模式)      │
                    └────────┬─────────────────┘
                             │
        ┌────────────────────┼────────────────────┐
        │                    │                    │
  ┌─────▼──────┐    ┌───────▼───────┐    ┌───────▼───────┐
  │   Stage 1  │    │    Stage 2    │    │    Stage 3    │
  │ Data Prep  │    │ Analysis Plan │    │ Analysis Exec │
  └─────┬──────┘    └───────┬───────┘    └───────┬───────┘
        │                    │                    │
  ┌─────▼──────┐    ┌───────▼───────┐    ┌───────▼───────┐
  │    Gate    │    │     Gate      │    │     Gate      │
  │   1.5      │    │     2.5       │    │     3.5       │
  │ (子Agent)  │    │   (子Agent)   │    │   (子Agent)   │
  └────────────┘    └───────────────┘    └───────────────┘
```

| Agent | 实现方式 | 理由 |
|-------|---------|------|
| **Pipeline Orchestrator** | 主对话Skill | 必须保持单一控制源 |
| **Data Validator** | 子Agent（可选） | 数据验证有明确边界，可独立执行 |
| **Method Consultant** | 子Agent（可选） | SAP制定需要深度统计学知识 |
| **Exec Runner** | 主对话内 | 代码生成-执行-自愈是紧密耦合循环 |
| **Exec Inference** | 子Agent（推荐） | Generator-Evaluator需要独立上下文 |
| **QC Inspector** | 子Agent（推荐） | 纯只读审查，可与下一阶段并行 |

### 4.3 具体实施路径

**Phase 1: QC Inspector子Agent化（优先级最高）**
- 实现：将Gate 1.5/2.5/3.5的检查逻辑封装为子Agent prompt
- 输入：产物文件路径 + 检查清单
- 输出：质量门闸报告（PASS/WARN/FAIL）
- 收益：Gate检查可与下一阶段准备并行

**Phase 2: Exec Inference子Agent化**
- 实现：将Phase 7-9的推断检验封装为子Agent
- 输入：分析结果文件 + SAP
- 输出：推断验证报告 + Generator-Evaluator差异
- 收益：真正的独立验证，上下文隔离

**Phase 3: Data Validator子Agent化（可选）**
- 实现：将数据验证逻辑封装为子Agent
- 输入：原始数据文件 + 数据字典
- 输出：清洗后数据 + 验证报告
- 收益：数据质量检查可与因果探索并行

### 5.2 关键重构方案

#### 方案A：质量门闸系统独立化

**当前状态**：质量门闸检查逻辑分散在各SKILL.md中，以Markdown检查清单形式存在。

**重构目标**：将质量门闸抽取为独立模块，支持Skill模式和子Agent模式双运行。

```
shared/quality-gates/
├── __init__.py
├── gate_runner.py          # 门闸执行器（支持Skill/子Agent双模式）
├── gates/
│   ├── gate_1_5_data.py    # Stage 1.5 数据质量门闸
│   ├── gate_2_5_sap.py     # Stage 2.5 SAP质量门闸
│   ├── gate_3_5_results.py # Stage 3.5 结果质量门闸
│   ├── gate_5_5_integrity.py  # Stage 5.5 审前完整性
│   └── gate_5_8_final.py   # Stage 5.8 最终完整性
├── checklists/             # 检查清单（Markdown）
│   ├── data_quality_9items.md
│   ├── sap_quality_8items.md
│   └── results_quality_9items.md
└── reports/                # 门闸报告模板
    └── ...
```

**实现方式**：
- `gate_runner.py` 提供统一的门闸执行接口
- 每个gate文件定义检查项、通过标准、判定逻辑
- 支持两种运行模式：Skill模式（LLM按清单执行）和子Agent模式（独立Agent执行）

#### 方案B：统计方法知识库统一索引

**当前状态**：统计方法引用分散在3个位置。

**重构目标**：创建统一索引，各处引用指向索引。

## 七、结论与建议

### 7.1 核心优势（保持）

1. **知识库深度**：48章统计方法指南 + 80+代码模板 + 16个报告规范检查清单，这是MSRA的核心竞争力
2. **质量门闸设计**：5个阻断式门闸 + Material Passport，确保分析质量
3. **人机回环**：关键决策点必须用户批准，避免AI自主决策风险
4. **研究类型分支**：RCT/观察性/诊断试验的完整方法学覆盖

### 7.2 关键改进建议

1. **短期(1-2周)**：修复版本号不一致、清理文档残留引用、对齐合约Schema
2. **中期(1-2月)**：实施QC Inspector子Agent化、创建统计方法统一索引、增强Pipeline集成测试
3. **长期(3-6月)**：实施Exec Inference子Agent化、推进8项优化计划、大规模数据E2E验证

### 7.3 架构演进方向

```
当前: Skill调度模式(模式A) — LLM扮演角色，上下文共享
  ↓
近期: 混合模式(模式C) — 编排层Skill + 关键角色子Agent
  ↓
远期: Python Agent模式(模式B) — 独立服务化（如需要）
```

**建议保持模式C作为默认运行模式**，因为它在隔离性和易用性之间取得了最佳平衡。模式B仅在需要独立服务化时才考虑。
