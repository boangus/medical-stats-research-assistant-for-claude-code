> **版本**: v1.0.0 | **更新日期**: 2026-06-25 | **状态**: 已完成

> **前置文档**: `00-项目总览与架构.md`、`28-架构评估与重构建议.md`、`MSRA-v1.0-optimization-report.md`

## 1. 评估背景

MSRA v1.0.0 已正式发布（2026-06-26），包含 13 个 Skill、16 个命令、5 个阻断式质量门闸、6 个实验性模块门闸、319 个测试全部通过。

本次评估聚焦两个核心问题：

1. **技能配置冗长** — 4 个核心 SKILL.md 超过 1000 行，每次加载消耗大量 context window，降低 LLM 推理质量
2. **流程架构可拆分性** — 识别可模块化或分离的组件，提出简化和重构的具体建议

### 1.1 评估方法

- 逐文件行数统计 + 依赖关系矩阵分析
- 有界上下文识别（Domain-Driven Design 方法）
- 与已验证的 data-prep 瘦身模式（991→236行，-76%）对标
- 架构决策记录（ADR）格式记录关键决策

## 4. 技能配置瘦身方案

### 4.1 已验证模式：data-prep 的 phases/ 抽取

data-prep 是唯一完成瘦身的核心 Skill，其模式可推广：

```
瘦身前: SKILL.md = 991 行（所有 Phase 内联）
瘦身后: SKILL.md = 236 行 + phases/ × 8 文件 ≈ 700 行
效果: 主文件 -76%，加载时间显著降低，Phase 规范独立可维护
```

**抽取规则**:
- 主 SKILL.md 保留：角色定义 + 架构图 + Phase 索引表 + 检查点汇总 + 快速参考
- phases/ 目录存放：每个 Phase 的完整执行规范（输入/输出/步骤/Checkpoint/异常处理）
- shared/ 目录存放：跨 Phase 复用的规则、模板、约束

### 4.2 瘦身目标与路径

## 5. 流程架构可拆分性分析

### 5.1 当前架构：单体 Pipeline

```
/msra → pipeline/SKILL.md (785行)
  → data-prep/SKILL.md → 8 phase files
  → analysis-plan/SKILL.md → 6 phase files
  → analysis-exec/SKILL.md → 7 phase files
  → report/SKILL.md → phase files
```

**问题**: Pipeline SKILL.md 是唯一的编排入口，785 行的编排逻辑与质量门闸定义混合。

### 5.2 建议架构：编排器 + 独立 Skills

```
┌──────────────────────────────────────────────────┐
│  Layer 0: Pipeline Orchestrator (精简至 ~300行)    │
│  职责: 意图检测 → 阶段路由 → 进度追踪 → Checkpoint  │
│  不包含: 质量门闸检查项、Phase 详细规范              │
└───────────────────────┬──────────────────────────┘
                        │ dispatch
        ┌───────────────┼───────────────┐
        ▼               ▼               ▼
┌──────────────┐ ┌──────────────┐ ┌──────────────┐
│ Stage Skills │ │ Gate Skills  │ │ Extension    │
│ (独立运行)    │ │ (独立运行)    │ │ Modules      │
│              │ │              │ │              │
│ data-prep    │ │ gate-data    │ │ bioinformatics│
│ analysis-plan│ │ gate-sap     │ │ imaging      │
│ analysis-exec│ │ gate-results │ │ realtime     │
│ report       │ │              │ │ cross-domain │
└──────────────┘ └──────────────┘ └──────────────┘
        │               │
        ▼               ▼
┌──────────────────────────────────────────────────┐
│  Layer 2: Shared Kernel                           │
│  statistics-methods/ | quality_gates/ | templates/ │
│  sap/ | anti-patterns/ | reporting-guidelines/     │
└──────────────────────────────────────────────────┘
```

### 5.3 可拆分组件清单

| 组件 | 当前位置 | 建议位置 | 可拆分性 | 理由 |
|------|---------|---------|---------|------|
| 质量门闸定义 | pipeline/SKILL.md L196-335 | shared/quality_gates/{gate-name}.md | ✅ 完全可拆 | 门闸是独立检查器，仅需输入/输出接口 |

### 5.4 不建议拆分的组件

| 组件 | 原因 |
|------|------|
| data-prep phases/ | 已完成瘦身，结构良好 |

## 6. 架构决策记录（ADR）

### ADR-001: 技能配置瘦身策略

**Status**: Proposed

**Context**: 4 个核心 SKILL.md 超过 1000 行，消耗大量 context window，降低 LLM 推理质量。data-prep 已验证 phases/ 抽取模式有效（991→236行，-76%）。

**Decision**: 对 analysis-exec、analysis-plan、report、calibration 4 个 Skill 执行与 data-prep 相同的瘦身策略：
1. Phase 详细规范抽取到 phases/ 目录
2. 共享规则抽取到 shared/ 对应子目录
3. 主 SKILL.md 仅保留：角色定义 + 架构图 + Phase 索引表 + 检查点汇总 + 快速参考

**Consequences**:
- ✅ 主文件加载行数减少 70-80%
- ✅ Phase 规范独立可维护，修改不影响主文件
- ⚠️ 需要更新所有内部引用路径
- ⚠️ 需要确保 Claude Code 能正确解析 phases/ 子目录中的 Markdown 文件

### ADR-002: 质量门闸独立化

**Status**: Proposed

**Context**: 质量门闸定义分散在 pipeline/SKILL.md（Gate 1.5/2.5/3.5）和各 Stage SKILL.md 中，修改门闸需要同时更新多个文件，存在同步风险。

**Decision**: 将所有质量门闸抽取为 shared/quality_gates/ 下的独立文件，Pipeline 通过引用加载。

**Consequences**:
- ✅ 门闸定义唯一数据源，消除同步问题
- ✅ 门闸可被扩展模块（bio/img/rt/cross）复用
- ⚠️ 需要定义标准化的门闸接口（输入/输出/判定规则）

### ADR-003: Agent 角色定义单一来源

**Status**: Proposed

**Context**: Agent 角色行为在 agents/*.md 和各 SKILL.md 中双重定义（如 analysis-exec 中 Exec Runner 的行为规范与 agents/exec_runner_agent.md 可能不同步）。

**Decision**: agents/*.md 为唯一定义源，SKILL.md 仅通过引用链接指向 Agent 定义，不内联角色行为规范。

**Consequences**:
- ✅ 消除双重定义的同步风险
- ✅ Agent 角色可独立演进
- ⚠️ SKILL.md 中的 Agent 速查表需改为引用链接
