# 20. 合约系统设计

## 20.1 概述

MSRA 合约系统是定义在 `shared/contracts/` 下的 JSON Schema 集合，用于规范跨 Skill、跨阶段的数据交换和质量保证。合约系统是 MSRA 质量门闸体系的基础设施，确保每个阶段的输出符合下游阶段的预期。

### 20.1.1 合约架构

```
shared/contracts/
├── README.md                    # 合约系统文档
├── sprint_contract.schema.json  # Sprint 合约 Schema (Schema 13)
├── passport/                    # Passport 子 Schema
│   ├── data_preparation.json
│   ├── analysis_plan.json
│   ├── analysis_execution.json
│   ├── calibration_result.json
│   ├── manuscript_draft.json
│   └── report_generation.json
├── audit/                       # Audit 子 Schema
│   ├── audit_jsonl.schema.json
│   ├── audit_sidecar.schema.json
│   └── audit_verdict.schema.json
├── evaluator/                   # Evaluator 合约
│   └── full.json
└── reviewer/                    # Reviewer 合约
    ├── full.json
    └── methodology_focus.json
```

### 20.1.2 合约类型

| 类型 | 用途 | 版本 |
|------|------|------|
| **Sprint 合约** | 定义 Reviewer/Evaluator 的评审标准和失败条件 | v3.6.2+ |
| **Passport 合约** | 定义各阶段的输入/输出产物规范 | v3.6.4+ |
| **Audit 合约** | 定义跨模型审计的证据格式 | v3.6.7+ |

## 20.2 Sprint 合约 (sprint_contract.schema.json)

### 20.2.1 Schema 定义

Sprint 合约基于 Schema 13.1，定义了评审面板的组成、评估维度和失败条件。

```json
{
  "$schema": "https://json-schema.org/draft/2020-12/schema",
  "$id": "msra/sprint_contract/v13.1",
  "title": "MSRA Sprint Contract",
  "description": "Schema for reviewer/evaluator sprint contracts",
  "type": "object",
  "required": ["panel_size", "acceptance_dimensions", "failure_conditions"],
  "properties": {
    "panel_size": {
      "type": "integer",
      "minimum": 1,
      "description": "评审面板人数"
    },
    "acceptance_dimensions": {
      "type": "array",
      "items": {
        "type": "object",
        "required": ["dimension_id", "name", "weight"],
        "properties": {
          "dimension_id": {"type": "string"},
          "name": {"type": "string"},
          "weight": {"type": "number", "minimum": 0, "maximum": 1}
        }
      }
    },
    "failure_conditions": {
      "type": "array",
      "items": {
        "type": "object",
        "required": ["condition_id", "severity", "action"],
        "properties": {
          "condition_id": {"type": "string"},
          "severity": {"enum": ["critical", "major", "minor", "info"]},
          "action": {"enum": ["reject", "revise", "warn", "skip"]},
          "expression": {"type": "string"}
        }
      }
    }
  }
}
```

### 20.2.2 使用示例

**Reviewer 完整合约 (reviewer/full.json)**

```json
{
  "contract_id": "msra/reviewer/full/v1",
  "mode": "reviewer_full",
  "panel_size": 5,
  "acceptance_dimensions": [
    {"dimension_id": "D1", "name": "方法学严谨性", "weight": 0.25},
    {"dimension_id": "D2", "name": "结果呈现完整性", "weight": 0.20},
    {"dimension_id": "D3", "name": "讨论深度", "weight": 0.20},
    {"dimension_id": "D4", "name": "引用质量", "weight": 0.15},
    {"dimension_id": "D5", "name": "语言规范性", "weight": 0.20}
  ],
  "failure_conditions": [
    {"condition_id": "F1", "severity": "critical", "action": "reject",
     "expression": "D1 < 0.6"},
    {"condition_id": "F2", "severity": "major", "action": "revise",
     "expression": "any_dimension < 0.7"},
    {"condition_id": "F3", "severity": "major", "action": "revise",
     "expression": "total_score < 0.75"},
    {"condition_id": "F4", "severity": "minor", "action": "warn",
     "expression": "D5 < 0.8"}
  ]
}
```

### 20.2.3 验证流程

```bash
# 验证合约文件
python scripts/check_sprint_contract.py shared/contracts/reviewer/full.json

# 验证所有合约
python scripts/check_sprint_contract.py shared/contracts/**/*.json

# 指定 MSRA 版本验证
python scripts/check_sprint_contract.py contract.json --MSRA-version v3.6.2
```

## 20.3 Passport 子 Schema (passport/)

### 20.3.1 概述

Passport 合约定义了 MSRA 各阶段的输入/输出产物规范，是 Material Passport 系统的 Schema 基础。

### 20.3.2 data_preparation.json — 数据准备阶段

**用途**: 定义 Stage 1 数据准备阶段的产物规范

**必需产物**

| artifact_id | 名称 | 说明 |
|-------------|------|------|
| raw_data | 原始数据 | 原始未修改的数据文件 |
| data_dictionary | 数据字典 | 变量名、标签、类型、编码方案 |
| cleaning_log | 清洗日志 | 所有清洗操作的记录 |
| analysis_dataset | 分析数据集 | 清洗后可供分析的数据集 |
| variable_listing | 变量清单 | 所有变量的完整清单 |
| missing_data_summary | 缺失数据摘要 | 缺失模式和比例 |
| demographic_summary | 基线摘要 | 分析人群的人口统计学特征 |

**质量门闸**

| gate_id | 名称 | 说明 |
|---------|------|------|
| DG-01 | 数据完整性检查 | 无重复记录，唯一标识有效 |
| DG-02 | 变量范围检查 | 所有值在预期范围内 |
| DG-03 | 缺失模式检查 | 无意外的系统性缺失 |
| DG-04 | 分析人群一致性 | ITT/PP/SAFETY 人群定义一致 |

### 20.3.3 analysis_plan.json — 分析计划阶段

**用途**: 定义 Stage 2 分析计划阶段的产物规范

**必需产物**

| artifact_id | 名称 | 说明 |
|-------------|------|------|
| sap | 统计分析计划 | 完整的 SAP 文档 |
| method_selection_rationale | 方法选择依据 | 每个统计方法的选择理由 |
| sample_size_calculation | 样本量计算 | 样本量计算过程和结果 |
| multiplicity_adjustment | 多重性调整 | 多重比较的调整策略 |

### 20.3.4 analysis_execution.json — 分析执行阶段

**用途**: 定义 Stage 3 分析执行阶段的产物规范

**必需产物**

| artifact_id | 名称 | 说明 |
|-------------|------|------|
| analysis_code | 分析代码 | R/Python 分析脚本 |
| analysis_results | 分析结果 | 统计结果（CSV/JSON） |
| diagnostic_plots | 诊断图 | 模型诊断图 |
| sensitivity_analysis | 敏感性分析 | 敏感性分析结果 |

### 20.3.5 calibration_result.json — 校准结果

**用途**: 定义跨模型校准的产物规范

**必需产物**

| artifact_id | 名称 | 说明 |
|-------------|------|------|
| calibration_report | 校准报告 | 校准过程和结果 |
| cross_model_comparison | 跨模型对比 | 不同模型的结果对比 |
| consensus_method | 共识方法 | 达成共识的方法 |

### 20.3.6 manuscript_draft.json — 稿件草稿

**用途**: 定义论文写作阶段的产物规范

**必需产物**

| artifact_id | 名称 | 说明 |
|-------------|------|------|
| manuscript_draft | 稿件草稿 | 论文初稿 |
| references | 参考文献 | 引用文献列表 |
| supplementary_materials | 补充材料 | 补充表格、图形 |

### 20.3.7 report_generation.json — 报告生成阶段

**用途**: 定义最终报告生成的产物规范

**必需产物**

| artifact_id | 名称 | 说明 |
|-------------|------|------|
| final_report | 最终报告 | 完整的统计分析报告 |
| executive_summary | 执行摘要 | 报告摘要 |
| tables_figures | 表格与图形 | 所有表格和图形 |
| compliance_report | 合规报告 | 报告规范合规检查结果 |

## 20.4 Audit 子 Schema (audit/)

### 20.4.1 概述

Audit 合约定义了跨模型审计的证据格式，用于记录和验证 AI Agent 的审计过程。

### 20.4.2 audit_jsonl.schema.json — JSONL 事件流

**用途**: 定义 Codex CLI 的 JSON 事件流格式

**事件类型**

| event_type | 说明 | 必需字段 |
|------------|------|----------|
| thread.started | 线程开始 | thread_id, timestamp |
| turn.started | 轮次开始 | turn_id, thread_id, timestamp |
| item.completed | 项目完成 | item_id, turn_id, content, timestamp |
| turn.completed | 轮次完成 | turn_id, status, timestamp |
| error | 错误 | error_code, message, timestamp |

**示例**

```json
{"event_type": "thread.started", "thread_id": "t-001", "timestamp": "2026-06-23T10:00:00Z"}
{"event_type": "turn.started", "turn_id": "tu-001", "thread_id": "t-001", "timestamp": "2026-06-23T10:00:01Z"}
{"event_type": "item.completed", "item_id": "i-001", "turn_id": "tu-001", "content": "...", "timestamp": "2026-06-23T10:00:05Z"}
{"event_type": "turn.completed", "turn_id": "tu-001", "status": "success", "timestamp": "2026-06-23T10:00:10Z"}
```

### 20.4.3 audit_sidecar.schema.json — 元数据旁路

**用途**: 定义审计运行的元数据格式

**字段定义**

| 字段 | 类型 | 说明 |
|------|------|------|
| runner | object | 运行器信息（名称、版本） |
| timing | object | 时间信息（开始、结束、持续） |
| process | object | 进程信息（PID、内存、CPU） |
| stream | object | 流信息（输入/输出路径） |
| prompt | object | 提示词信息（模板、变量） |

### 20.4.4 audit_verdict.schema.json — 审计裁决

**用途**: 定义审计裁决文件的格式

**裁决类型**

| verdict | 说明 | 后续动作 |
|---------|------|----------|
| PASS | 通过 | 允许继续 |
| MINOR | 轻微问题 | 记录警告，允许继续 |
| MATERIAL | 重大问题 | 需要人工审查 |
| AUDIT_FAILED | 审计失败 | 阻断流程 |

**示例**

```json
{
  "verdict": "MINOR",
  "finding_counts": {
    "minor": 2,
    "major": 0,
    "critical": 0
  },
  "failure_reason": null,
  "timestamp": "2026-06-23T10:05:00Z",
  "reviewer": "codex-audit-v1"
}
```

## 20.5 Evaluator 子 Schema

### 20.5.1 概述

Evaluator 合约定义了评估 Agent 的评审标准和评分计划。

### 20.5.2 evaluator/full.json — 完整评估合约

**评估维度**

| dimension_id | 名称 | 权重 | 说明 |
|--------------|------|------|------|
| D1 | 原创性 | 0.20 | 研究问题的新颖性 |
| D2 | 方法学严谨性 | 0.25 | 统计方法的适当性 |
| D3 | 证据充分性 | 0.20 | 结果支持结论的程度 |
| D4 | 论证连贯性 | 0.15 | 逻辑推理的严密性 |
| D5 | 写作质量 | 0.20 | 语言表达的规范性 |

**失败条件**

| condition_id | severity | action | 表达式 |
|--------------|----------|--------|--------|
| F1 | critical | reject | D2 < 0.5 |
| F2 | critical | reject | D3 < 0.5 |
| F3 | major | revise | any_dimension < 0.6 |
| F4 | major | revise | total_score < 0.65 |
| F5 | minor | warn | D5 < 0.7 |
| F6 | minor | warn | D1 < 0.6 |
| F0 | - | accept | all_dimensions >= 0.7 AND total_score >= 0.75 |

**评分计划 (scoring_plan)**

```json
{
  "scoring_plan": {
    "scale": "0-1",
    "precision": 2,
    "aggregation": "weighted_mean",
    "calibration_required": true,
    "cross_reviewer_variance_threshold": 0.15
  }
}
```

## 20.7 合约验证流程

### 20.7.1 验证时机

1. **开发阶段**: 添加新合约时必须验证
2. **CI/CD**: 自动化测试中验证所有合约
3. **运行时**: Agent 启动时验证合约有效性
4. **门闸检查**: Gate 检查时验证产物是否符合合约

### 20.7.2 验证步骤

```
合约验证流程
    │
    ▼
[1] Schema 语法验证
    │ - JSON 格式正确
    │ - 必需字段存在
    │ - 字段类型正确
    │
    ▼
[2] 语义验证
    │ - 枚举值有效
    │ - 数值范围合理
    │ - 表达式语法正确
    │
    ▼
[3] 一致性验证
    │ - 权重总和为 1
    │ - dimension_id 唯一
    │ - condition_id 唯一
    │
    ▼
[4] 版本兼容性验证
    │ - 合约版本与 MSRA 版本兼容
    │ - Schema 版本与验证器兼容
    │
    ▼
验证完成
```

### 20.7.3 验证工具

```bash
# 验证单个合约
python scripts/check_sprint_contract.py <contract.json>

# 验证所有合约
python scripts/check_sprint_contract.py shared/contracts/**/*.json

# 在代码中验证
from shared.contracts import validate_contract

result = validate_contract("shared/contracts/reviewer/full.json")
if result.is_valid:
    print("合约有效")
else:
    print(f"验证失败: {result.errors}")
```

### 20.7.4 添加新合约

1. 在 `shared/contracts/<domain>/` 下创建 JSON 文件
2. 运行验证脚本确认无错误
3. 更新 `shared/contracts/README.md` 文档
4. 在相关 Agent 中引用新合约
5. 添加测试用例
