## 1. 项目概述

### 1.1 定位

Medical Statistics Research Assistant (MSRA) 是一个 **Claude Code 插件**，以 Skill + Command 形式运行，提供 Pipeline 驱动的医学统计分析 + 学术写作能力，覆盖从原始数据到可投稿论文的完整研究流程。

**插件形态**: Claude Code Plugin（通过 `manifest.json` 注册命令，通过 `skills/` 和 `commands/` 定义功能）

### 1.2 设计目标

| 目标 | 说明 |
|------|------|
| **端到端覆盖** | 数据输入 → 清洗 → 分析 → 报告 → 论文，全流程自动化 |
| **人机回环** | 关键决策点（清洗策略、方法选择、SAP确认）必须用户批准 |
| **可重复性** | 所有步骤生成可重复代码，随机种子/数据版本/环境完整记录 |
| **方案遵循** | SAP 一经确认，分析严格按计划执行，偏差必须走变更管理 |
| **质量门闸** | 5 个阻断式质量门闸，未通过不放行 |

### 1.3 目标用户

- 医学/中医药研究生、临床研究者
- 需要完成课题数据分析但缺乏专业统计支持的研究人员
- 需要从数据到论文一站式完成的科研工作者

### 1.4 插件架构

MSRA 作为 Claude Code 插件，遵循以下架构约束：

```
manifest.json (插件清单)
    │  注册 12 个命令 (/msra, /msra-data, /msra-plan, ...)
    │  定义入口点 (entry_point → SKILL.md)
    ▼
commands/ (命令定义)
    │  12 个 .md 文件，每个对应一个 /msra-* 命令
    │  定义命令的触发条件、参数、用法示例
    ▼
skills/ (技能定义)
    │  9 个 SKILL.md，每个定义一个完整的功能模块
    │  包含：角色定义、处理流程、IRON RULES、质量门闸
    │  由 LLM 加载并执行（LLM 扮演对应角色）
    ▼
shared/ (共享资源)
    │  模板、契约、方法指南、报告规范
    │  被 skills 引用，提供领域知识
    ▼
agents/ (Agent 框架 - 参考实现)
    │  Python 代码定义的 Agent 接口和实现
    │  当前未被 Pipeline 实际调用，作为架构规范和未来演进
    ▼
tests/ (测试套件)
       50+ 测试文件，覆盖 E2E / 单元 / 集成
```

**插件运行机制**:
1. 用户在 Claude Code 中输入 `/msra` 等命令
2. Claude Code 根据 `manifest.json` 找到对应的 `entry_point` (SKILL.md)
3. SKILL.md 的内容被加载到 LLM 上下文中
4. LLM 按 SKILL.md 定义的角色、规则、流程执行任务
5. 跨阶段信息通过 Pipeline 编排器在上下文中传递

## 3. Pipeline 流水线设计

### 3.1 阶段划分

| Stage | 名称 | Skill/命令 | 核心职责 |
|-------|------|-----------|---------|
| **Stage 1** | 数据准备 | `/msra-data` → `data-prep` | 数据验证 → 清洗 → 值规范化 → EDA → 盲态审核 → 数据库锁定 |
| **Stage 1.5** | 数据质量门闸 | 内置阻断检查 | 9 项数据质量检查，未通过退回 Stage 1 |
| **Stage 1.8** | 探索性因果分析 | `/msra-explore` → `exploratory-causal` | 因果发现 → 混杂识别 → 假设生成（可选，无先验假设时执行） |
| **Stage 2** | 分析计划 | `/msra-plan` → `analysis-plan` | EDA → Estimands → 方法探讨 → SAP → 审查 → 定稿 |
| **Stage 2.5** | SAP 质量门闸 | 内置阻断检查 | 8 项 SAP 完整性检查，未通过退回 Stage 2 |
| **Stage 3** | 分析执行 | `/msra-exec` → `analysis-exec` | 变量构造 → 描述统计 → 主要分析 → 敏感性分析 → 质检 |
| **Stage 3.5** | 结果质量门闸 | 内置阻断检查 | 9 项结果完整性检查，未通过退回 Stage 3 |
| **Stage 4** | 报告生成 | `/msra-report` → `report` | 结果解读 → 表格/图表 → 方法学 → 规范检查 |
| **Checkpoint** | 分支决策 | 用户交互 | [A] 仅报告 / [B] 继续写论文 |
| **Stage 5** | 论文写作 | `/msra-paper` → `report` | 文献检索 → IMRaD 撰写 → 评审 → 修订 → 定稿 |
| **Stage 6** | 过程总结 | 内置 | 完整过程记录 + AI 自我反思 |

### 3.2 质量门闸体系

```
Stage 1 ──→ 🔴 Gate 1.5 (9项) ──→ Stage 2 ──→ 🔴 Gate 2.5 (8项)
──→ Stage 3 ──→ 🔴 Gate 3.5 (9项) ──→ Stage 4

Stage 5.5 (审前完整性) ──→ Stage 5.8 (最终完整性)
```

| 门闸 | 检查项数 | 检查内容 | 失败处理 |
|------|---------|---------|---------|
| **Gate 1.5** | 9 项 | 数据完整性、缺失率、异常值、变量类型、PHI检测 | 退回 Stage 1 |
| **Gate 2.5** | 8 项 | SAP 完整性、Estimands 五要素、方法匹配、变量构造逻辑 | 退回 Stage 2 |
| **Gate 3.5** | 9 项 | 结果完整性、P值合理性、图表质量、代码可复现 | 退回 Stage 3 |
| **Gate 5.5** | — | 论文完整性检查 | 退回修订 |
| **Gate 5.8** | — | 最终完整性检查 | 退回修订 |

### 3.3 人机回环节点

| 节点 | 交互内容 | 必须确认 |
|------|---------|---------|
| 数据清洗策略 | 清洗方案讨论 → 用户批准 | ✅ |
| 研究方向确认 | 探索结果 → 用户选择方向 | ✅ |
| SAP 制定 | 方法探讨 → SAP 框架 → 用户审查定稿 | ✅ |
| 变量名确认 | 标准化变量名清单 → 用户确认 | ✅ |
| Checkpoint A/B | 仅报告 / 继续写论文 | ✅ |
| 变更管理 | 偏离 SAP 时 → 提交变更请求 → 用户确认 | ✅ |

## 4. Multi-Agent 架构

### 4.1 Agent 角色定义

| Agent | 角色 | 对应 Skill | 核心职责 |
|-------|------|-----------|---------|
| **Pipeline Orchestrator** | 编排器 | `pipeline` | 意图检测、阶段路由、进度追踪、人机回环 |
| **Data Validator** | 数据质量专家 | `data-prep` | 数据验证、清洗策略、变量构造、盲态审核 |
| **Method Consultant** | 资深生物统计学家 | `analysis-plan` | EDA、Estimands、方法选择、SAP 制定与审查 |
| **Exec Runner** | 统计程序员 | `analysis-exec` (Phase 0-6) | 代码生成、自愈 Debug（5轮）、[SKIP] 标记 |
| **Exec Inference** | 统计推断专家 | `analysis-exec` (Phase 7-9) | 独立假设检验、质量检查、Generator-Evaluator 比对 |
| **QC Inspector** | 质量审查员 | 跨阶段 | 质量门闸执行、一致性检查、偏差检测、结果复现 |
| **Arbitrator** | 仲裁者 | 内置 (ConflictResolver) | 冲突仲裁、投票决策、升级处理 |

### 4.2 核心基础设施

| 组件 | 文件 | 功能 |
|------|------|------|
| **IAgent Interface** | `agents/core/interfaces.py` | 标准 Agent 接口（initialize/execute/send_message/receive_message） |
| **BaseAgent** | `agents/core/base_agent.py` | Agent 基类，提供通用生命周期管理 |
| **MessageBus** | `agents/core/communication.py` | 异步消息队列，支持优先级和确认机制 |
| **TaskQueue** | `agents/core/communication.py` | 优先级任务队列，支持依赖管理 |
| **MultiLevelCache** | `agents/core/cache.py` | L1(内存) + L2(磁盘) 多级缓存 |
| **ConflictResolver** | `agents/core/conflict_resolution.py` | 冲突检测与解决，支持仲裁机制 |
| **Registry** | `agents/core/registry.py` | Agent 注册与发现 |

### 4.3 协作模式

```
Spec → Review → Build → Test

Stage 1: Orchestrator → Data Validator (构建数据)
                      → QC Inspector 审查 (Gate 1.5)

Stage 2: Orchestrator → Method Consultant (制定SAP)
                      → QC Inspector 审查 (Gate 2.5)

Stage 3: Orchestrator → Exec Runner (生成代码)
                      → Exec Inference (推断检验)
                      → QC Inspector 审查 (Gate 3.5)

Stage 4: Orchestrator → Report Expert (生成报告)
```

### 4.4 混合模式架构（Skill + 子 Agent）

MSRA 支持三种运行模式，**模式 C（混合模式）** 是推荐的默认模式：

| 模式 | 说明 | 适用场景 |
|------|------|---------|
| **A: Skill 调度** | 所有角色在同一上下文中扮演 | 快速原型、调试 |
| **B: Python Agent** | 独立进程，MessageBus 通信 | 高并发/多用户（未来） |
| **C: 混合模式** ⭐ | 编排层 Skill + 关键角色拆分为子 Agent | 日常使用（推荐） |

**混合模式下的 Agent 拆分**:

| Agent | 拆分方式 | 核心收益 |
|-------|---------|---------|
| **QC Inspector** | ✅ 子 Agent | Gate 检查与下一阶段准备并行；只看产物不看过程，检查更独立 |
| **Exec Inference** | ✅ 子 Agent | 真正独立于 Exec Runner，Generator-Evaluator 比对有意义 |
| **Data Validator** | ✅ 子 Agent | 数据验证可与因果探索并行；PHI 敏感数据上下文隔离 |
| **Method Consultant** | ✅ 子 Agent | 独立上下文加载更多方法学参考 |
| **Exec Runner** | ❌ 主对话 | 代码生成-执行-自愈是紧耦合循环 |
| **Pipeline Orchestrator** | ❌ 主对话 | 必须保持单一控制源 |

> 详见 `docs/dev/07-Agent框架设计.md` §7 混合模式架构。

## 5. 跨阶段数据契约 (Handoff Contracts)

### 5.1 Schema 清单

| Schema | 名称 | 生产者 | 消费者 |
|--------|------|--------|--------|
| Schema 1 | PICO/PICOS 研究问题定义 | data-prep / analysis-plan | analysis-plan / report |
| Schema 2 | SAP Elements & Estimands | analysis-plan | analysis-exec / calibration / report |
| Schema 3 | 数据画像报告 | data-prep | analysis-plan / pipeline |
| Schema 4 | 清洗日志 | data-prep | analysis-exec / pipeline |
| Schema 5 | 分析结果摘要 | analysis-exec | report / peer-review |
| Schema 6 | 质量门闸报告 | QC Inspector | pipeline / 各阶段 |
| Schema 7 | Material Passport | 各阶段 | 下游阶段 |
| Schema 8 | 变更请求 | 任意阶段 | pipeline (审批) |

### 5.2 交接验证规则

- 所有 schema 采用 Markdown 结构化输出
- 消费方 agent 必须在接收时验证所有必填字段
- 缺失必填字段触发 `HANDOFF_INCOMPLETE` 失败路径
- Schema 违规要求重新生成

## 6. 语言一致性约束

### 6.1 核心规则

> **IRON RULE**: 分析语言必须在整个流水线中保持一致。选定 R 则全程 R，选定 Python 则全程 Python，禁止混用。

### 6.2 实现机制

1. **Stage 2 SAP 阶段锁定语言**：SAP 中明确记录 `analysis_language: R | Python`
2. **Stage 3 执行阶段强制检查**：Exec Runner 根据 SAP 锁定的语言生成代码
3. **Gate 3.5 语言一致性检查**：QC Inspector 检查所有生成代码的语言一致性
4. **违规处理**：发现语言混用 → 阻断 → 退回 Stage 3 修正

| **运行平台** | Claude Code | Skill 形式运行 |
| **统计语言** | R / Python | 用户选定后全程一致 |
| **R 包** | tidyverse, gtsummary, survival, rms, tableone | 核心统计 |
| **Python 包** | pandas, numpy, scipy, statsmodels, lifelines, sklearn | 核心统计 |
| **大规模处理** | Polars / DuckDB / Dask | 按数据规模自动选择 |
| **报告格式** | HTML + Markdown + PNG + DOCX | 多格式输出 |

### 7.2 大规模处理引擎选择策略

| 数据规模 | 推荐引擎 | 切换阈值 | 说明 |
|----------|---------|---------|------|
| <10K 行 | pandas (Python) / base R | 默认 | 内存足够，无需额外引擎 |
| 10K-100K 行 | Polars (Python) | 自动切换 | 列式存储，惰性求值，内存效率高 |
| 100K-1M 行 | DuckDB | 自动切换 | 内嵌 OLAP，SQL 接口，跨语言 |
| >1M 行 | Dask | 自动切换 | 分布式计算，支持超出内存的数据集 |

引擎选择由 `shared/large_scale_processing/engine_selector.py` 自动完成，用户无需手动指定。切换逻辑基于 `engine_factory.py` 的 `select_engine(data_size, memory_limit)` 接口。
