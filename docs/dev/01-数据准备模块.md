# 数据准备模块 (Data Preparation) — Stage 1

> **Skill**: `skills/data-prep/` | **命令**: `/msra-data` | **版本**: v0.9.0

---

## 1. 模块概述

数据准备模块是 MSRA Pipeline 的第一个阶段，负责在分析前确保数据的完整性和正确性。核心设计原则是 **"先验证、再讨论、获批准后执行"**，永远不会自动清洗数据。

### 1.1 角色定位

- **角色**: 医学数据质量专家
- **输入**: 原始数据文件（CSV/Excel/多文件目录）
- **输出**: 清洗后数据 + 清洗日志 + 验证报告 + EDA报告 + 盲态审核记录 + 锁定记录 + 规范化日志

**高级数据源支持**:

| 数据源 | 支持状态 | 说明 |
|--------|---------|------|
| CSV/Excel | ✅ 直接支持 | 默认输入格式 |
| 多文件目录 | ✅ 直接支持 | 自动检测同结构文件 |
| Parquet | ✅ 通过引擎 | 由 Polars/DuckDB 引擎处理 |
| 数据库连接 | 🔧 实验性 | 通过 DuckDB SQL 接口 |

### 1.2 混合模式下的 Data Validator

在混合模式（模式 C）下，Data Validator 可作为独立子 Agent 运行：

| 维度 | Skill 模式 | 混合模式 |
|------|-----------|---------|
| **验证上下文** | 与编排器共享 | 独立上下文 |
| **并行能力** | 串行 | 数据验证 ∥ 因果探索准备 |
| **PHI 检测** | 主对话内 | 独立上下文（敏感数据隔离） |

**适用场景**:
- 大型数据集的验证和清洗（独立上下文避免占用主对话窗口）
- 需要同时启动探索性因果分析时（并行执行）
- PHI 敏感数据处理（隔离上下文更安全）

**不适用场景**:
- 需要频繁与用户交互讨论清洗策略时（子 Agent 交互能力受限）
- 小型数据集的快速清洗（启动子 Agent 有额外开销）

**降级**: 子 Agent 不可用时，在主对话内以 Skill 模式执行。

### 1.3 Iron Rules

1. 永远不自动清洗数据——总是先验证、再讨论、获得批准后才执行
2. 静默删除任何数据是绝对禁止的——删除操作必须用户明确确认
3. 值规范化日志必须完整记录 原值→规范值 映射，供门闸消费
4. 盲法试验数据审核必须在盲态下进行

## 2. 处理流程

```
输入数据
    │
    ▼
Phase 0.5: 多数据集模式检测
    │  检测单文件/多文件/多时间点模式
    ▼
Phase 1: 数据画像与验证 (MANDATORY)
    │  ├─ Step 0: 快速画像 (1页概览)
    │  ├─ 结构/类型/缺失/逻辑/范围检查
    │  ├─ CDISC/SDTM 合规检查
    │  └─ PHI 隐私检测
    ▼
Phase 2: 清洗与规范化策略 (MANDATORY)
    │  ├─ 清洗策略讨论（交互式）
    │  ├─ Step 2.5: 值规范化（检测到变体时执行）
    │  └─ TCM 术语变体 (GB/T 16751) + 数值格式变体
    ▼
Phase 3: 执行清洗 + 清洗日志
    │  按批准的策略执行，完整记录每一步操作
    ▼
Phase 4: EDA 质量检查
    │  清洗后数据的探索性分析
    ▼
Phase 5: 盲态审核 + 数据库锁定 (MANDATORY)
    │  盲法试验数据在盲态下审核，锁定后数据不可修改
    ▼
Phase 6: 可复现性验证 (CONDITIONAL)
    │  含随机性操作时，自动重跑3次验证一致性
    │  由 shared/reproducibility/ 模块支持
    ▼
Phase 7: 数据质量门闸 (9项阻断检查)
    │  → Pipeline Stage 1.5 消费
    ▼
输出: 清洗后数据 + 完整文档包
```

## 3. 各 Phase 详细设计

### 3.1 Phase 0.5: 多数据集模式检测

**目的**: 检测输入数据的组织模式，决定后续处理策略。

| 模式 | 检测条件 | 处理策略 |
|------|---------|---------|
| 单文件 | 单个 CSV/Excel | 直接加载 |
| 多文件目录 | 目录下多个同结构文件 | 合并策略讨论 |
| 多时间点 | 检测到时间维度 | 纵向数据处理 |

### 3.2 Phase 1: 数据画像与验证

**Step 0: 快速画像**

生成 1 页数据概览：
- 行列数、变量类型分布
- 缺失率热图
- 基本统计摘要（数值变量: 中位数[IQR]；分类变量: n(%)）

**结构检查**

| 检查项 | 说明 | 失败处理 |
|--------|------|---------|
| 列名规范 | 检查特殊字符、空格、中文混用 | 标记需修正 |
| 数据类型 | 数值/分类/日期类型是否正确 | 标记需转换 |
| 重复行 | 完全重复或主键重复 | 标记需去重 |
| 缺失模式 | MCAR/MAR/MNAR 初步判断 | 影响清洗策略 |

**逻辑检查**

- 范围检查：数值是否在合理范围内
- 一致性检查：相关变量间逻辑是否一致
- 时序检查：日期变量的时间顺序是否合理

**PHI 隐私检测**

基于 HIPAA Safe Harbor 18 类标识符自动扫描：
- 姓名、地址、日期（年份>89岁）、电话、传真、邮箱
- SSN、病历号、健康计划号、账号、车牌、设备标识
- URL、IP、生物标识、照片、唯一编码

### 3.3 Phase 2: 清洗与规范化策略

**交互式清洗策略讨论**

必须与用户讨论并确认以下决策：

| 决策项 | 选项 | 说明 |
|--------|------|------|
| 缺失值处理 | 删除/均值填充/MICE/多重插补 | 根据缺失机制选择 |
| 异常值处理 | 保留/截尾/删除 | 必须说明理由 |
| 变量转换 | 对数/标准化/分箱 | 需要文献或理论依据 |
| 重编码 | 分类合并/参考水平 | 需要临床意义 |

**值规范化 (Step 2.5)**

检测并规范化以下变体：
- TCM 术语变体（GB/T 16751 标准）
- 数值格式变体（单位、精度、编码）
- 分类变量水平变体（空格、大小写、同义词）

### 3.4 Phase 3: 执行清洗

- 按批准的策略逐项执行
- 每步操作记录到清洗日志（原值→新值→理由→操作人）
- 清洗日志格式遵循 `shared/templates/data-prep/cleaning_log_template.md`

### 3.5 Phase 4: EDA 质量检查

清洗后数据的探索性分析：
- 分布检查（正态性、偏度、峰度）
- 相关性矩阵
- 缺失值残余检查
- 数据质量评分

### 3.6 Phase 5: 盲态审核 + 数据库锁定

- **盲态审核**: 盲法试验中，审核者不知道分组信息
- **数据库锁定**: 锁定后数据不可修改，任何修改需要解锁流程
- SOP 遵循 `shared/templates/data-prep/data_lock_sop_template.md`

### 3.7 Phase 6: 可复现性验证（条件触发）

当数据准备过程中涉及以下操作时自动触发：
- 随机抽样（train/test 拆分等）
- 多重插补（MICE 等）
- Bootstrap 重采样

验证方式：使用 `shared/reproducibility/reproducibility_check.py` 自动重跑 3 次，检查关键输出的一致性。

> **注意**: Phase 编号中 5→7 的跳跃是历史设计遗留（Phase 6 为后期新增的条件阶段），后续版本将统一编号。

## 4. 数据质量门闸 (Gate 1.5)

9 项阻断式检查，全部通过才能进入 Stage 2：

| # | 检查项 | 通过标准 | 失败处理 |
|---|--------|---------|---------|
| 1 | 数据完整性 | 所有必填变量无缺失 | 退回补充 |
| 2 | 缺失率 | 核心变量缺失率 <5% | 退回处理 |
| 3 | 异常值 | 无未处理的极端异常值 | 退回确认 |
| 4 | 变量类型 | 所有变量类型正确 | 退回修正 |
| 5 | 逻辑一致性 | 无逻辑矛盾 | 退回修正 |
| 6 | 清洗日志 | 清洗日志完整 | 退回补充 |
| 7 | PHI 检测 | 无未脱敏的 PHI | 退回处理 |
| 8 | 盲态审核 | 盲法试验已完成盲态审核 | 退回执行 |
| 9 | 数据库锁定 | 数据库已锁定 | 退回锁定 |
