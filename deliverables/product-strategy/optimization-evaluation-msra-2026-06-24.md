## 1. 产品目标

### 目标 1：降低首次使用门槛
新用户从安装到完成第一次分析的时间应控制在 30 分钟以内。当前瓶颈是依赖安装体积大（29 个核心依赖含 PyMC/Dask）、16 个命令缺乏引导。

### 目标 2：优化 Claude Code 上下文效率
核心 SKILL.md 的 token 消耗需要降低 40%+，确保在 Claude Code 单次对话中能完整加载并执行，不触发上下文截断。

### 目标 3：增强分析流程的实用性和可信度
数据画像、变量可视化、结果解读等环节的增强，让用户对分析过程有更强的掌控感和信任感。

## 2. 用户故事

### 场景 1：新用户首次使用
> 作为医学研究者，我希望安装 MSRA 后能快速了解它的能力，通过一个简单的示例数据完成第一次分析，以便判断它是否适合我的研究需求。

**痛点**：当前 16 个命令 + 14 个 Skill 对新用户有认知负担，缺少"快速上手"引导。

### 场景 2：数据上传后的初步了解
> 作为临床研究者，上传数据后我希望先看到数据的整体画像（变量数、缺失率、分布概要），再决定如何进行清洗和分析。

**痛点**：当前 data-prep 直接进入验证检查，缺少全局概览步骤。

### 场景 3：复杂衍生变量的理解
> 作为多中心 RCT 的统计师，我希望直观看到变量之间的依赖关系 DAG，以便确认衍生变量的推导链路是否正确。

**痛点**：当前变量构建公式以 Markdown 表格形式呈现，不直观。

### 场景 4：统计报告到论文的过渡
> 作为论文作者，我希望从统计报告中自动提取结构化的论文素材（核心发现、安全性数据、局限性），以便高效撰写论文。

**痛点**：当前 handoff bundle 提取的内容不够全面，缺少安全性发现和局限性。

### 场景 5：多中心数据整合
> 作为多中心研究的协调者，我希望同时输入多个中心的数据集，系统自动检查中心间一致性并进行汇总分析。

**痛点**：当前仅支持单数据集，多中心研究需要手动合并。

## 5. 新增优化建议

### 5.1 SKILL.md 上下文瘦身（P0）

**问题**：4 个核心 SKILL.md 超过 1,000 行（analysis-exec 1,250 行、report 1,229 行、analysis-plan 1,132 行、data-prep 991 行），Claude Code 单次对话上下文压力大。

**方案**：将 SKILL.md 中的详细 Phase 规范抽取为独立引用文件（如 `skills/analysis-exec/phases/phase1-validation.md`），SKILL.md 仅保留目录索引和关键规则。

**验收标准**：
- 每个核心 SKILL.md 主文件控制在 500 行以内
- 详细 Phase 规范独立存放，可通过索引快速定位
- 所有现有功能不受影响

**工作量**：中（~200 行编辑 + 文件重组）

### 5.2 依赖分层与安装优化（P0）

**问题**：29 个核心依赖包含 PyMC（安装重）、Dask（依赖链复杂）、CVXPY（仅为综合控制模板引入），首次安装可能耗时 10+ 分钟。

**方案**：
- 将 `pymc`、`dask`、`cvxpy` 移至可选依赖组
- 创建 `requirements-core.txt`（精简版，~15 个依赖）和 `requirements-full.txt`（完整版）
- 安装脚本支持 `--minimal` 和 `--full` 两种模式

**验收标准**：
- `pip install msra --minimal` 安装时间 < 3 分钟
- 核心功能（data-prep/analysis-plan/analysis-exec/report）在 minimal 模式下完全可用
- 文档明确说明各可选依赖的使用场景

**工作量**：中（~100 行编辑 + 文档更新）

### 5.3 渐进式学习引导（P1）

**问题**：16 个命令 + 14 个 Skill 对新用户有认知负担，缺少"快速上手"路径。

**方案**：
- 创建 `/msra-quickstart` 命令，提供交互式引导
- 新增 `shared/templates/quickstart_tutorial.py`，使用内置示例数据
- 在 `/msra` 命令中增加"首次使用？试试 /msra-quickstart"提示

**验收标准**：
- `/msra-quickstart` 在 10 分钟内引导用户完成一次完整分析
- 示例数据内置，无需用户准备
- 完成后推荐下一步学习路径

**工作量**：中（~300 行新建 + 文档更新）

### 5.4 CI/CD 集成（P1）

**问题**：`.github/workflows/ci.yml` 已配置（3 job：smoke-test/lint/security），但仅 Python 3.13 单版本，缺少多版本矩阵测试和覆盖率 badge。

**方案**：
- 扩展 CI 配置为 Python 3.9/3.10/3.11/3.13 矩阵测试
- 添加 pytest-cov 覆盖率报告上传到 Codecov
- 添加 badge 到 README

**验收标准**：
- 每次 push 自动运行 4 个 Python 版本的测试矩阵
- PR 必须通过所有测试才能合并
- README 显示测试状态 + 覆盖率 badge

**工作量**：小（~30 行 CI 配置编辑）

### 5.5 README 数字一致性修正（P0）

**问题**：README 声称统计方法 47-48 章，实际 `shared/statistics-methods/` 下仅 13 个 .md 文件。

**方案**：核实实际章节数，统一 README 中的数字描述。

**验收标准**：
- README 中的数字与实际文件数一致
- 如有合并章节，明确说明合并逻辑

**工作量**：极小（~10 行编辑）

## 8. Non-goals（明确不做什么）

- **不重写核心架构**：现有 passport/quality gates/agent 框架成熟，不做大重构
- **不增加新的医学统计方法**：当前覆盖已全面，新方法通过模板扩展而非核心修改
- **不做 GUI/可视化界面**：保持 Claude Code 原生交互，不引入 Web UI
- **不做多语言支持**：保持中英双语，不扩展其他语言
- **不做云端部署**：保持本地插件定位，不做 SaaS 化
- **不做实时协作**：保持单用户模式，不做多人协作功能
