## 📌 TL;DR（执行摘要）

- **核心目标**：评估 MSRA（Medical Statistics Research Assistant）Claude Code 插件在医学统计分析工具市场中的竞争地位和产品价值
- **关键决策**：MSRA 在"AI + 统计流水线"细分领域具有先发优势，但需在易用性、社区建设和文档体系上持续投入
- **下一步**：优先降低学习曲线（P0）、补充核心模板测试（P0）、建设中文用户社区（P1）

## 3. 市场数据与趋势

### 3.1 市场规模

| 市场细分 | 规模（2025 估计） | CAGR | 驱动因素 |
|----------|-------------------|------|----------|
| 医学统计软件 | $25 亿 | 8.2% | 临床试验增长、循证医学普及 |
| AI 辅助研究工具 | $8 亿 | 22.5% | LLM 能力提升、研究效率需求 |
| 生物信息学平台 | $12 亿 | 15.3% | 精准医疗、组学数据爆发 |
| 学术出版工具 | $5 亿 | 6.8% | 开放获取运动、出版合规要求 |

### 3.2 关键趋势

1. **AI 原生统计工具崛起** — 2025-2026 年出现大量 AI 辅助数据分析工具，但多数聚焦通用场景，医学专用极少
2. **可重复性成为刚需** — ICMJE、NIH、Wellcome Trust 均要求数据和代码共享，MSRA 的 Passport 机制契合此趋势
3. **质量门闸理念扩散** — FDA 对 AI/ML 医疗器械的审查趋严，统计分析质量控制需求上升
4. **多模态融合** — 影像 + 基因组 + 临床数据的联合分析成为前沿，MSRA 的 cross-domain 模块有先发优势
5. **Claude Code 生态扩展** — 插件/Skill 机制降低了分发门槛，但也意味着竞品进入成本低

### 3.3 用户画像数据

| 维度 | 数据 |
|------|------|
| 目标用户群规模（中国） | 医学硕博研究生 ~30 万/年，临床研究者 ~10 万 |
| 核心痛点排序 | ① 统计方法选择 > ② 报告规范合规 > ③ 可重复性 > ④ 编程能力不足 |
| 现有工具分布 | SPSS（45%）> R（25%）> SAS（15%）> Stata（10%）> 其他（5%） |
| AI 工具接受度 | 65% 愿意尝试 AI 辅助统计，但仅 15% 信任 AI 直接输出结果 |
| 付费意愿 | 月均 $10-30（学生），$50-100（机构），更倾向免费/开源 |

## 4. 用户感知（来自瑞思）

### 4.1 核心痛点被精准定位

MSRA 针对医学研究人员的统计方法选择困难、报告规范复杂、可重复性要求高等痛点，设计了 13 个 Skill 和 5 个阻断式质量门闸，表明产品对用户需求有深入理解。

### 4.2 用户满意度预测

**用户可能喜欢的特性：**
- ✅ 自动化统计报告生成 — 符合 CONSORT/STROBE 等规范
- ✅ 质量保障机制 — 阻断式门闸防止统计错误
- ✅ 本地数据处理 — 满足医学数据隐私要求
- ✅ 可重复性保障 — Pipeline 设计确保分析可追溯

**用户可能不满的方面：**
- ❌ 学习曲线陡峭 — 16 个命令对新手造成认知负担
- ❌ 技术门槛 — 仍需基本 R/Python 编程能力
- ❌ 错误信息不友好 — 技术性错误提示可能困惑非技术用户
- ❌ 缺乏 GUI — 纯命令行模式限制了用户群体

### 4.3 关键用户假设待验证

- 用户是否真正需要 AI 辅助统计分析？（65% 愿意尝试）
- 用户对质量门闸机制的接受程度？（需实际测试）
- 用户愿意投入多少时间学习新工具？（关键阈值：2 周内能独立使用）

## 5. SWOT 分析

### Strengths（优势）

| 维度 | 说明 |
|------|------|
| **Pipeline 架构** | 7 阶段 + 5 个阻断式门闸，调度器不做实质性工作，架构干净 |
| **统计方法覆盖** | 56 章统计方法指南，从基础 t 检验到孟德尔随机化，覆盖全面 |
| **报告规范体系** | 21 个报告规范 + 5 个偏倚评估工具，SCI 论文合规率高 |
| **双语言支持** | R + Python 满足不同技术背景用户 |
| **扩展模块** | 生物信息学/医学影像/实时分析/跨领域融合，4 模块 319 测试全通过 |
| **本地处理** | 数据不出本机，满足 HIPAA / 医院信息安全要求 |
| **开源免费** | MIT 许可证（代码）+ CC BY-NC-SA（知识库），降低使用门槛 |

### Weaknesses（劣势）

| 维度 | 说明 |
|------|------|
| **学习曲线陡峭** | 16 个命令 + Pipeline 概念，新手上手困难 |
| **无 GUI** | 纯命令行/CLI 模式，限制了非技术用户群体 |
| **社区薄弱** | 缺乏活跃的中文用户社区和案例分享 |
| **测试覆盖不均** | 核心模板缺少单元测试（27 个模板仅少数有测试） |
| **文档偏向开发者** | 用户友好指南不足，中文教程匮乏 |
| **TemplateManager 注册不全** | 27 个模板仅 12 个注册到映射表 |
| **SKILL.md 臃肿** | 部分主文件超过 1000 行，LLM 上下文浪费 |

### Opportunities（机会）

| 维度 | 说明 |
|------|------|
| **AI 统计工具蓝海** | 医学专用 AI 统计工具极少，MSRA 有先发优势 |
| **可重复性需求增长** | ICMJE/NIH 要求代码共享，Passport 机制契合趋势 |
| **多模态融合前沿** | 影像 + 基因组 + 临床数据联合分析是前沿方向 |
| **中医药研究特色** | 中医临床研究的统计需求（证候量化、TCM 变量编码）是差异化切入点 |
| **Claude Code 生态** | 插件机制降低分发门槛，可快速触达 Claude 用户 |
| **研究生群体刚需** | 中国 ~30 万/年医学硕博有统计分析需求，且对免费工具有强需求 |

### Threats（威胁）

| 维度 | 说明 |
|------|------|
| **大厂 AI 统计工具** | ChatGPT Code Interpreter、Julius AI 等通用 AI 分析工具可能蚕食市场 |
| **RStudio/Posit AI 化** | RStudio 正在集成 AI 功能（GitHub Copilot 集成），可能压缩 MSRA 空间 |
| **竞品进入成本低** | Claude Code 插件机制意味着任何人可开发类似 Skill |
| **用户信任壁垒** | 医学研究者对 AI 生成的统计结果持谨慎态度 |
| **维护成本高** | 56 章统计方法 + 21 个报告规范需要持续更新 |

## 6. 差异化机会

### 6.1 核心差异化定位

**"医学研究者的一站式 AI 统计助手"** — 不是通用数据分析工具，而是专为医学研究设计的端到端流水线。

### 6.2 差异化优势清单

| # | 差异化维度 | 竞品状态 | MSRA 独特性 |
|---|-----------|----------|------------|
| 1 | **阻断式质量门闸** | 无竞品有此机制 | 5 个门闸 + 清单式检查，不通过就退回 |
| 2 | **报告规范自动合规** | Prism 部分支持 | 21 个规范自动检查（CONSORT/STROBE/PRISMA...） |
| 3 | **从数据到论文的全链路** | 竞品仅覆盖部分阶段 | Stage 1-6 完整流水线 |
| 4 | **多模态融合分析** | 无竞品覆盖 | 影像 + 基因组 + 临床数据跨域分析 |
| 5 | **中医药特色支持** | 无竞品覆盖 | CONSORT-TCM/SPIRIT-TCM/STRICTA + 证候量化 |
| 6 | **Material Passport** | 无竞品有此机制 | 每步产物可追溯，满足可重复性要求 |

### 6.3 潜在蓝海

1. **中医药临床研究** — 全球首个支持 CONSORT-TCM/SPIRIT-TCM 的 AI 统计工具
2. **实施科学分析** — 新兴研究范式，现有工具几乎无覆盖
3. **因果推断流水线** — DAG + PSM/IPTW + TMLE/AIPW 的自动化编排
4. **适应性设计** — 临床试验中期分析、样本量重估的自动化

## ⚠️ 待确认 / 假设 / Non-goals

### 假设
- MSRA 的目标用户是医学研究人员（硕博/临床研究者），而非专业生物统计学家
- 用户有基本的 R 或 Python 编程能力（至少能运行脚本）
- Claude Code 的用户基数足够支撑 MSRA 的初期增长
- 免费开源模式可以吸引足够用户，后续通过增值服务变现

### 待确认
- [ ] 实际用户使用 MSRA 后的满意度和留存率
- [ ] 用户对质量门闸"阻断式"机制的接受程度（是否觉得过于严格？）
- [ ] Claude Code 插件的分发和发现机制是否足以支撑用户增长
- [ ] 中医药特色模块的实际市场需求有多大

### Non-goals
- ❌ 不追求成为通用数据分析平台（专注医学研究）
- ❌ 不追求 GUI 优先（保持 CLI + Pipeline 的核心架构）
- ❌ 不追求替代 SPSS/SAS（定位 AI 辅助层，而非替代传统工具）
- ❌ 不追求商业变现（当前阶段聚焦用户增长和社区建设）
