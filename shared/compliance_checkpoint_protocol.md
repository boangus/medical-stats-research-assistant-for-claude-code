# 临床研究合规检查协议

定义合规检查如何参与 Stage 2.5（分析计划合规）和 Stage 4.5（最终报告合规）的完整性门控，
以及决策如何与现有的 FAIL 循环交互，用户 override 如何处理。

**使用者**: `pipeline` skill (Stage 2.5 / 4.5)。

---

## 1. 适用法规框架

本协议覆盖以下临床研究合规框架：

| 框架 | 范围 | 适用场景 |
|------|------|----------|
| **ICH-GCP (E6 R2)** | 临床试验数据质量与受试者保护 | 所有临床试验 |
| **赫尔辛基宣言** | 人体研究伦理基本原则 | 所有涉及人的研究 |
| **ICH E9(R1)** | Estimand 框架与统计分析计划 | 主要/次要终点分析 |
| **CONSORT 2010** | 随机对照试验报告规范 | RCT 论文 |
| **STROBE** | 观察性研究报告规范 | 队列/病例对照/横断面研究 |
| **PRISMA 2020** | 系统综述/Meta 分析报告规范 | 系统综述 |
| **HIPAA** | 美国健康信息隐私保护 | 涉及美国受试者的数据 |
| **GDPR** | 欧盟通用数据保护条例 | 涉及欧盟受试者的数据 |
| **中国个人信息保护法** | 中国个人信息保护 | 涉及中国受试者的数据 |

---

## 2. 双门控概览

| 维度 | Stage 2.5（分析计划合规） | Stage 4.5（最终报告合规） |
|------|--------------------------|--------------------------|
| **检查时机** | SAP 定稿后、分析执行前 | 统计报告完成后、论文提交前 |
| **ICH-GCP 检查** | 协议合规性、知情同意、IRB 审批 | 数据完整性、安全性报告、审计追踪 |
| **统计方法合规** | Estimand 定义、多重性策略、样本量论证 | 分析执行与 SAP 一致性、敏感性分析 |
| **报告规范合规** | N/A（尚未生成报告） | CONSORT/STROBE/PRISMA 清单完成度 |
| **数据隐私合规** | 脱敏策略、数据访问控制 | 数据共享声明、匿名化验证 |
| **伦理合规** | 注册信息、利益冲突、DSMB 设置 | 不良事件报告、受试者保护证据 |
| **阻断条件** | 任何 ICH-GCP 强制条目 FAIL = 阻断 | 任何 ICH-GCP 或报告规范强制条目 FAIL = 阻断 |
| **警告条件** | 高度推荐条目 FAIL，或 RAISE 原则 = warn | 同左 |
| **信息条件** | 推荐/可选条目 FAIL | 同左 |
| **门控输出** | 完整检查点 + dashboard 中 `compliance` 部分 | 同左 |

## 3. 决策优先级

在单次检查点内，决策聚合为：

```
overall_decision = max_severity(
  gcp_decision,                    // ICH-GCP 合规
  statistical_method_decision,     // 统计方法合规 (E9/R1)
  reporting_guideline_decision,    // 报告规范合规 (CONSORT/STROBE/PRISMA)
  data_privacy_decision,           // 数据隐私合规 (HIPAA/GDPR)
  ethical_compliance_decision,     // 伦理合规
  legacy_integrity_decision        // 已有完整性验证结果
)

severity order: pass < info < warn < block
```

### 3.1 ICH-GCP 检查维度

| 维度 | GCP 条目 | Stage 2.5 | Stage 4.5 |
|------|----------|-----------|-----------|
| **知情同意** | 2.9, 4.8 | 检查知情同意流程描述 | 验证同意率、撤回处理 |
| **IRB/IEC 审批** | 3.2, 4.4 | 检查 IRB 审批信息完整性 | 验证各中心 IRB 审批号 |
| **研究注册** | — | 检查注册号、注册日期 | 验证注册信息与实际一致 |
| **方案偏离** | 4.5, 5.18 | 检查方案偏离记录计划 | 报告方案偏离发生率及影响 |
| **数据完整性** | 5.1, 5.5 | 检查数据管理计划 | 验证电子数据采集 (EDC) 审计追踪 |
| **安全性报告** | 4.11, 5.17 | 检查 AE/SAE 报告流程 | 验证 AE 汇总、SUSAR 报告 |
| **监查计划** | 5.18.1 | 检查监查计划描述 | 验证监查报告（如适用） |
| **统计分析** | 5.5.2 | 检查 SAP 与 E9(R1) 一致性 | 验证分析执行与 SAP 一致 |

### 3.2 统计方法合规（ICH E9(R1)）

| 条目 | 检查内容 | 严重级别 |
|------|----------|----------|
| **Estimand 定义** | 每个主要/次要终点必须有完整 estimand 定义 | Mandatory |
| **缺失数据策略** | 必须指定主要分析和敏感性分析的缺失数据处理方法 | Mandatory |
| **多重性控制** | 涉及多个主要终点时必须有正式的多重性控制策略 | Mandatory |
| **样本量论证** | 必须有基于临床意义的样本量论证 | Mandatory |
| **分析人群定义** | ITT、mITT、PP、Safety 人群必须有明确的纳入/排除标准 | Mandatory |
| **中间分析** | 如有中期分析，必须有预设的停止规则和 alpha 消耗函数 | Mandatory (if applicable) |
| **软件验证** | 统计软件版本及包信息必须完整记录 | Highly Recommended |
| **代码可重复性** | 分析代码必须可提供审阅 | Highly Recommended |

### 3.3 报告规范合规

| 报告规范 | 适用研究类型 | 关键条目（Stage 4.5 检查） |
|----------|-------------|--------------------------|
| **CONSORT** | RCT | 流程图、ITT 分析、随机化方法、盲法、基线表、主要结果 CI |
| **STROBE** | 观察性研究 | 研究设计、环境、参与者、变量、数据源、统计方法、敏感性分析 |
| **PRISMA** | 系统综述 | 检索策略、PRISMA 流程图、偏倚评估、证据质量 |
| **STARD** | 诊断准确性研究 | 预设阈值、参考标准、敏感度/特异度、ROC 曲线 |
| **TRIPOD** | 预测模型 | 模型开发 vs 验证、校准曲线、临床效用分析 |

### 3.4 数据隐私合规

| 条目 | Stage 2.5 | Stage 4.5 |
|------|-----------|-----------|
| **脱敏策略** | 检查数据脱敏方案（k-匿名性、差分隐私等） | 验证输出数据匿名化 |
| **数据访问控制** | 检查角色权限矩阵 | 验证数据访问日志 |
| **数据保留** | 检查数据保留期限计划 | 验证保留策略执行 |
| **跨境传输** | 如涉及跨境数据，检查传输合规方案 | 验证传输记录 |
| **数据共享声明** | N/A | 验证论文中包含数据共享声明 |

## 4. 合规检查执行流程

### 4.1 Stage 2.5 — 分析计划合规检查

```
SAP 完成
  ↓
自动合规检查器启动
  ↓
┌─ ICH-GCP 检查 ─────────────────┐
│ - 知情同意流程描述               │
│ - IRB 审批信息                   │
│ - 研究注册号                     │
│ - 安全性报告流程                 │
│ - 数据管理计划                   │
└─────────────────────────────────┘
  ↓
┌─ E9(R1) 统计方法检查 ──────────┐
│ - Estimand 完整性               │
│ - 缺失数据策略                  │
│ - 多重性控制                    │
│ - 样本量论证                    │
│ - 分析人群定义                  │
└─────────────────────────────────┘
  ↓
┌─ 数据隐私检查 ──────────────────┐
│ - 脱敏方案                      │
│ - 数据访问控制                  │
│ - 数据保留计划                  │
└─────────────────────────────────┘
  ↓
聚合决策 → overall_decision
  ├── pass → 进入 Stage 3（分析执行）
  ├── info → 仪表板提示，继续
  ├── warn → 仪表板警告，用户确认后继续
  └── block → 阻断，必须修复后重新检查
```

### 4.2 Stage 4.5 — 最终报告合规检查

```
统计报告/论文完成
  ↓
自动合规检查器启动
  ↓
┌─ ICH-GCP 深度检查 ─────────────┐
│ - 数据完整性验证                │
│ - 安全性报告完整性              │
│ - 方案偏离报告                  │
│ - 审计追踪                      │
└─────────────────────────────────┘
  ↓
┌─ 报告规范清单检查 ──────────────┐
│ - CONSORT/STROBE/PRISMA 清单   │
│ - 每个条目合规状态              │
│ - 缺失条目列表                  │
└─────────────────────────────────┘
  ↓
┌─ 统计一致性检查 ────────────────┐
│ - 报告结果与 SAP 一致           │
│ - 敏感性分析执行                │
│ - 亚组分析预设                  │
│ - 软件版本记录                  │
└─────────────────────────────────┘
  ↓
┌─ 最终数据隐私检查 ──────────────┐
│ - 输出数据匿名化验证            │
│ - 数据共享声明                  │
│ - 补充材料数据检查              │
└─────────────────────────────────┘
  ↓
聚合决策 → overall_decision
  ├── pass → 可提交/发布
  ├── info → 仪表板提示
  ├── warn → 仪表板警告，需确认
  └── block → 必须修复后重新检查
```

## 5. 用户操作 UX

### 5.1 阻断时

Orchestrator 展示：

```
[COMPLIANCE BLOCK] Stage <N>

GCP 检查失败: IRB 审批信息缺失
E9(R1) 检查失败: Estimand 未完整定义
报告规范检查失败: CONSORT Item 17a 未报告

请选择:
  (a) 返回补充 — 返回相应阶段补充缺失材料
  (b) 手动处理 — 直接编辑文档，重新运行合规检查
  (c) 承认局限 — 调用用户 override（见 §6 Override 梯度）
```

### 5.2 警告时

仪表板提示，完整检查点。用户以自由文本确认或输入 `continue`。

### 5.3 信息提示时（推荐/可选条目失败）

仪表板单条提示，不中断流程。

## 6. Override 梯度（3 轮摩擦机制）

当用户在阻断时选择"承认局限"触发。

| 轮次 | 行为 |
|------|------|
| **第 1 次 override** | 警告用户。理由可选。允许。 |
| **第 2 次** | 要求提供理由字符串（任意长度）。 |
| **第 3 次** | 要求理由 ≥ 100 字符。仅在理由确认后释放。 |

轮次计数按 stage × pipeline run 存储在 `compliance_history[].user_override` 中。

> **执行边界**：此梯度由 **pipeline skill 在运行时** 通过 `compliance_history[]` 轮次计数器执行，
> 而非通过 Schema。Schema 有意允许 `user_override.rationale.minLength: 1`，
> 以便历史 passport 和跨会话恢复不会因历史条目而验证失败。

### 6.1 `disclosure_addendum` 模板

每次成功 override，agent 生成 `disclosure_addendum` 文本并由 orchestrator **自动注入**到论文的 AI 使用声明部分。该 addendum 不可移除。

```
The authors acknowledge the following compliance limitations:
- ICH-GCP <item_id> (<item_title>) not fully addressed. Rationale: <user_text>.
- Statistical method <item_id>: <partial|not met>. Rationale: <user_text>.
- Reporting guideline <item_id> (<item_title>) not fully reported. Rationale: <user_text>.
```

### 6.2 填充规则

- `<item_id>`: 合规检查报告中的条目 ID
- `<item_title>`: 条目标题（如 "IRB approval" → "IRB Approval", "Estimand" → "Estimand Definition"）
- `<partial|not met>`: 条目原状态为 `warn` 时选 `partial`；为 `block` 时选 `not met`
- `<user_text>`: `user_override.rationale` 的逐字复制，超过 500 字符时截断（完整理由存于 compliance_history）

## 7. FAIL 循环集成

同一阶段连续 3 轮未解决的 retry 后，orchestrator 启动 FAIL 循环：

1. 列出所有无法解决的合规问题
2. 用户选择 [手动处理 / 从此阶段移除 AI 使用 / 带"部分不合规"警告继续]
3. 选择的路径记录为 `compliance_history[].resolution_path`

### 7.1 Resolution Path 选项

| 路径 | 说明 | 后果 |
|------|------|------|
| `manual_fix` | 用户手动修改后重新检查 | 重新进入检查流程 |
| `remove_ai_usage` | 移除该阶段 AI 辅助的声明 | 论文中添加相应声明 |
| `proceed_with_warning` | 带警告继续 | 在报告中记录不合规声明 |
| `withdraw_study` | 中止研究 | 记录中止原因，停止流水线 |

## 8. 追加写入历史

`material_passport.compliance_history[]` 永远不覆盖。重试通过后，先前的失败条目保留并带有自己的时间戳。Stage 6 的 AI 自我反思报告引用完整历史以展示修正过程——这是 RAISE 透明度原则在合规检查中的应用。

---

## 9. 非流水线调用边界

当从独立写作 skill 调用时（无 pipeline 状态机）：

- 合规报告写入 `./compliance_reports/<ISO-timestamp>.yaml`（本地文件）
- 不触发流水线检查点仪表板
- `block_decision` 上限为 `warn`（不阻断）

---

## 10. 特殊场景处理

### 10.1 多中心研究

当研究涉及多个中心时：

- 每个中心的 IRB 审批必须独立列出
- 各中心知情同意流程差异必须说明
- 数据传输协议 (DTA) 必须记录

### 10.2 国际多中心研究

除多中心要求外：

- HIPAA 合规（美国中心）
- GDPR 合规（欧盟中心）
- 中国个人信息保护法合规（中国中心）
- 跨境数据传输协议

### 10.3 涉及弱势群体

当研究涉及儿童、孕妇、认知障碍患者等弱势群体时：

- 额外的伦理审查要求
- 代理知情同意流程
- 风险-收益评估强化

### 10.4 新药/器械临床试验

- IND/IDE 申请信息
- 数据安全监查委员会 (DSMB) 运作记录
- 研究者手册 (IB) 版本跟踪
- 药物警戒 (PV) 计划
